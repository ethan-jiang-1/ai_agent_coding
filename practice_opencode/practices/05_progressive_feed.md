# 05 渐进式信息投喂：先给锚点，再补文件和命令输出

## 核心习惯

OpenCode 支持 `@file`、`!command`、`--file` 和 custom commands，但这不等于你应该第一轮就把所有材料一口气灌满。

## 在 OpenCode 里怎么做

- 第一轮先给任务目标、关键约束和 2-5 个锚点文件。
- 第二轮再补日志、测试结果、最近 diff 或命令输出。
- 复杂任务把中间结论外化成文件，而不是一直依赖会话记忆。
- 提示词太长时，用 `/editor` 组织长输入，不要直接在主对话里贴成一大坨。
- 当你想观察工具链到底做了什么时，先用 `/details`，不要把“帮我解释你刚才都干了什么”混进主任务。

一个实用节奏：

1. 任务目标 + 关键约束
2. `@` 指几个核心文件
3. 再补 `!` 命令结果或 `--file` 附件
4. 如果任务分步，落一份计划或 handoff 文件
5. 下一轮基于文件继续

## 为什么这很重要

因为 OpenCode 会按你给的边界去读、去想、去调工具。边界清楚，它的自主检索才有价值；边界模糊，它就容易把无关材料一起带进来。

## 不要这样做

- 不要第一轮就贴满日志、架构文档和十几个文件内容。
- 不要把所有中间结论都留在会话里。
- 不要把“它能自己读文件”理解成“你不用给边界”。

## 验收标准

- 第一轮信息足够清晰，但不是信息大杂烩。
- 后续补充是按需要增加，而不是一开始塞满。
- 中间关键结论已经开始外化成文件或 custom command。
