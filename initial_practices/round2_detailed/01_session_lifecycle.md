# 第一章（深化）：会话生命周期——任务边界即会话边界

> 本章为 round1 同章的深化版，假设读者已熟悉 round1 内容。
> 标注：`A/B` — 本机 CLI 或官方文档核验；`I` — 工程推论

---

## 习惯

**任务边界即会话边界。不同工具对"会话"的定义不同，需要分别理解才能做出正确的重置决策。**

---

## 正确的做法

### 深化：各工具"新会话"的实际操作

round1 列出了基本操作，但 Codex CLI 的部分在 grounding audit 后有重要修正：

**Claude Code**（`A/B`：claude --help 核验）

```bash
exit                              # 结束当前会话
claude                            # 开启新会话

# 只有在明确要延续同一逻辑任务时，才使用：
claude --continue                 # 继续最近会话
claude --resume <session-id>      # 恢复特定会话
claude --resume <id> --fork-session  # 从历史会话分叉（新上下文，保留历史）
```

**Codex CLI**（`A/B`：本机 CLI 核验）

```bash
# 一次性任务（不需要可恢复的会话历史）
codex exec --ephemeral "完成任务 A"

# 可延续的任务（保留 resume 能力）
codex exec "第一步：分析架构"
codex resume --last "第二步：按分析结果实现"

# 不同于 Claude Code：每次 codex（不带 exec）默认是交互式会话，
# 退出后仍然可以 resume --last
```

**Codex CLI 的重要澄清**（`A/B`）：

round1 写过"Codex CLI 天然每次命令都是独立新会话"，这是不准确的。准确的说法是：
- `codex exec --ephemeral` → 临时会话，不保存，不可 resume
- `codex exec`（不加 --ephemeral）→ 会话保存到磁盘，可以 resume
- `codex`（交互式）→ 会话保存，可以 resume

Codex CLI 的"任务边界即会话边界"，意味着一次性批处理用 `--ephemeral`，跨步骤的复杂任务就不用 `--ephemeral`（保留 resume 能力）。

**Aider**（`A/B`：官方文档核验）

```bash
/clear        # 清空聊天历史（不保留），但保留当前文件范围（/add 的文件列表）
/exit         # 退出

# Aider 的特殊机制：聊天历史有 token 软上限
# 超过上限时自动开始摘要（--weak-model 负责摘要）
# 这意味着 Aider 长会话有自动的历史压缩机制，不需要完全依赖用户手动清空
```

### 深化：基于上下文质量的重置时机，而不只是时间

round1 提到"60-90 分钟"作为重置的时间参考。但更准确的信号是上下文质量的变化，而不是时间（`I`）：

| 信号 | 说明 | 建议动作 |
|------|------|---------|
| AI 开始引用已经被删除或重命名的文件/函数 | 上下文里的文件结构信息已过时 | 立即重置 |
| AI 反复给出你已明确否定过的方案 | 负向 few-shot 积累 | 立即重置 |
| AI 对同一个简单问题给出越来越复杂的答案 | 过度工程化偏见，上下文已被"高复杂度"历史污染 | 考虑重置 |
| AI 在同一文件的不同位置做了互相矛盾的修改 | 注意力分散，上下文过长 | 立即重置 |

## 错误的可能

### 深化：大杂烩会话的真正代价

round1 用"18,000 个 Token 的无关代码"来说明大杂烩会话的代价。这个数字是有来源的，但更重要的是理解**为什么**会发生这种情况（`I`）：

当上下文里同时包含"前端 CSS 修改历史"和"后端数据库重构"的讨论时，AI 在处理数据库问题时，其注意力机制会对上下文里的所有内容（包括 CSS 相关的代码）分配权重。这些权重虽然低，但会影响 AI 对"什么是相关信息"的判断，导致：
1. 检索到不相关的代码片段
2. 生成"融合"了两种上下文的奇怪输出

**Aider 的 repo map 对这个问题的部分缓解**（`A/B`）：

Aider 的图排名算法会根据当前聊天中**实际 /add 的文件**来计算哪些部分的 repo map 最相关。如果你清晰地 /add 了只和数据库相关的文件，repo map 会优先展示数据库相关的其他文件，而不是 CSS 相关内容。这在一定程度上对抗了大杂烩会话的问题。

但这只是缓解，不是根治——大杂烩的聊天历史里仍然包含 CSS 讨论的内容，repo map 只是过滤了文件索引部分。

## 背后的原理

### 深化：Aider 的自动历史摘要机制

round1 说"长会话成本呈二次方增长"，这对 Aider 来说需要补充说明（`A/B`）：

Aider 有聊天历史 token 软上限（`--chat-history-size` 配置，超过时 `--weak-model` 自动做摘要）。这意味着 Aider 的成本增长不完全是二次方的——当历史达到软上限时，摘要机制会把历史压缩，降低单轮增量成本。

但摘要本身也有代价：
1. 摘要是有损的，某些历史细节会丢失
2. 摘要任务本身消耗 token（由 --weak-model 处理）
3. 被摘要的历史已经发生了"context rot"，摘要并不能完全解决问题

所以 Aider 长会话的实际行为是：成本增长 → 软上限触发摘要 → 成本降低 → 再次增长 → 循环。不是纯粹的二次方，但也不是线性的。

## 延伸的习惯

### 深化：Codex CLI exec resume 链的设计原则

当一个复杂任务需要多个 exec 步骤时，exec resume 链是保留上下文连续性的方式（`A/B + I`）：

```bash
# 步骤 1：分析（只读）
codex exec -s read-only "分析 src/auth/ 的现有架构，总结关键接口"

# 步骤 2：规划（只读，resume 步骤 1 的历史）
codex exec resume --last -s read-only "基于刚才的分析，给出 OAuth2 迁移的完整计划"

# 步骤 3：实现（可写，resume 步骤 2 的历史）
codex exec resume --last -s workspace-write "按计划实现中间件层"

# 步骤 4：验证（只读，resume 步骤 3 的历史）
codex exec resume --last -s read-only "运行测试，报告结果"
```

这种模式的关键点（`I`）：
- 每个步骤都可以单独审查（resume --last 后检查输出）
- 如果某一步出问题，可以 fork 从那一步重来，不需要从头开始
- sandbox 可以在步骤间切换（规划用 read-only，执行用 workspace-write）

## 个人 / 协作注意

**个人使用**：理解自己常用工具的会话保存行为，不要依赖错误的默认假设。Claude Code 默认保存会话（可 resume）；Codex CLI exec 默认保存（除非加 --ephemeral）；Aider 保存聊天历史文件。

**团队协作**：
- 在 Claude Code 里，同一个项目目录下的所有人共享 auto-memory。如果团队成员 A 的一个混乱会话让 AI 记录了错误的"项目经验"，团队成员 B 启动新会话时可能受到影响。
- 建议约定：发现 auto-memory（MEMORY.md）里有错误内容时，及时清理或修正，而不是忽略。

## 延伸调查方向

1. **量化 context rot 的临界点**：目前无可靠公开数据，待实测验证。工程推论（`I`）：临界点更多取决于任务类型而非文件数量——跨文件一致性要求高的任务（如重构）比单文件任务更早出现 context rot。

2. **工具差异（Aider repo map 对 context rot 的延缓效果）**：Aider 的 repo map 机制确实能减少"无关代码进入上下文"的问题（`A/B`），但延缓 context rot 多少，目前无量化数据。

3. **会话长度与成本的实测关系**：不同工具的成本增长曲线不同——Aider 因为有历史摘要机制，不是纯二次方；Claude Code 和 Codex CLI 更接近二次方（无自动摘要）。但具体数字需要实测。

4. **"三次纠错"的最优阈值**：目前无科学依据，是经验值（`I`）。工程推论：三次是合理的下界，但不同任务类型（探索性 vs 精确修复）可能有不同的最优阈值。
