# 第十一章：订阅额度陷阱——先分清你买的到底是什么

> 本章标注：`A/B + I`

> 本章刻意不把某个日期的套餐价格、额度数字、加价比例写死。计费页面变化快，本章只保留这轮已审计工具里能稳定成立的计费原则。

---

## 为什么这一章必须单独讲

前十章解决的是“怎么把 AI 用对”。这一章解决的是另一件同样重要的事：**怎么避免把错误的任务交给错误的计费层**。

很多团队的问题不是“AI 没效果”，而是：
- 本该用中档模型的任务，却长期跑在最贵一档
- 本该走只读规划的任务，却直接进入长时间自治执行
- 本该落到本地或轻量模型的任务，却消耗了订阅里的高价值额度

---

## 先分三种成本结构

### 1. 订阅/额度型

典型代表是 Cursor 这类带产品套餐、模式分层和模型路由的工具。

这类工具的核心风险不是“你看不到模型”，而是：
- 不同模式消耗额度的速度不同
- 高阶模式、远程 agent、长时间自治任务，通常比前台短对话更贵
- 产品页的套餐名、额度和附加规则变化很快

### 2. BYOK / 按量计费型

典型代表是 Aider 和 Codex CLI 这类更接近“你选模型，我来组织工作流”的工具。

优点是透明：
- 你知道自己用了什么模型
- 你知道是交互式还是批处理
- 你通常能把高价推理、低价编辑、本地模型分开

风险也很直接：
- 如果你长期默认最强模型，账单会直线上升
- 如果你把简单任务交给高自治流程，token 和工具调用会一起放大

### 3. 混合限制型

Claude Code 这类工具既有交互式 coding 体验，也可能叠加模型层、使用量层和自动化层的不同限制。

最危险的误区是把“能一直用”误解成“成本结构不重要”。

---

## Cursor：把高价值额度留给真正难的问题

这轮能安全写成事实的是：
- Cursor 有多种工作模式和规则入口
- 有前台 agent，也有 Background Agents
- Background Agents 是远程异步环境，不应和前台短对话按同一种成本心智来使用

更稳的预算纪律是：
- 日常探索优先用 `Ask` 或默认模式
- 只有在架构决策、跨文件重构、复杂排障时，才升级到更贵的模式或模型
- 任何涉及远程 agent、长时间运行、自动装依赖和自动执行命令的流程，都要当成“高消耗工作流”来预算

不要把某个日期的套餐价格、积分池大小或附加费率写成长期事实。读者真正该记住的是：**Cursor 的高阶工作模式应当被视为稀缺预算，而不是默认入口。**

---

## Claude Code：模型、权限模式和自动化深度都会改变账单

Claude Code 这一轮有两个很重要的正式能力，和成本直接相关：
- 可以显式选 `--model`
- 非交互式 `--print` 支持 `--max-budget-usd`

这意味着你可以把“成本控制”前移成工作流设计，而不是事后看账单。

```bash
# 预算受控的非交互式调用
claude --print --max-budget-usd 2 --model sonnet "总结这批日志，只输出根因"
```

对 Claude Code，更稳的节流方式是：
- 规划阶段先用 `--permission-mode plan`
- 日常实现默认留在中档模型
- 真正需要深推理时再升级模型
- 长会话及时 `/compact`，不要把历史无限滚长

不要把某个社区传闻里的窗口大小、限流节奏或高峰时段乘数写成固定事实，除非当天重新核过官方文档。

---

## Aider：透明，但也最容易因为“默认最强”而超支

Aider 的好处是成本结构清楚：
- `--model` 决定主模型
- `--weak-model` 可以下沉轻任务
- `--editor-model` 可以把改写动作拆给更便宜的模型
- `--message`、`--read`、`--cache-prompts` 这些入口都很明确

因此，Aider 最值得执行的节约原则不是“少用”，而是“分工”：

```bash
aider \
  --model claude-sonnet-4-6 \
  --weak-model claude-haiku-4-5 \
  --editor-model claude-haiku-4-5 \
  --read docs/architecture.md \
  src/auth/login.ts
```

典型错误是：
- 主模型、弱模型、编辑模型全都用最贵的一档
- 连 README、注释、样板测试也走旗舰模型
- 不先用 `--read` 缩小参考范围，导致每轮都带大量无关上下文

---

## Codex CLI：自动化越强，越要显式设边界

Codex CLI 的成本控制点非常清楚：
- `--model`
- `exec` 与交互式会话的选择
- sandbox / approval 的自治程度
- `--oss` 本地 provider 入口

```bash
# 只读规划，避免直接进入昂贵的自治执行
codex exec --model gpt-5 --sandbox read-only "先给我迁移计划，不要动文件"

# 本地轻任务可下沉到 OSS provider
codex --oss "总结这段错误日志"
```

这里最需要避免的误解是：
- `--ephemeral` 不是“自动省钱模式”，它只是 session 是否落盘
- 有 `resume` / `fork` 不代表你就该长期把所有任务塞进一个会话
- 更强自治会提高吞吐，但也可能放大 token、命令执行和验证成本

Codex CLI 适合做“边界清楚、步骤清楚、预算清楚”的任务。越自动化，越要先把目标和范围写死。

---

## 混合工作流：把昂贵能力只留给高杠杆阶段

一个更稳的成本结构通常长这样：

```text
阶段 1：规划 / 诊断
- 用更强模型
- 产出计划文件

阶段 2：实现
- 回到中档模型
- 按计划改代码

阶段 3：检索 / 格式化 / 日志总结
- 下沉到轻量模型或本地模型
```

这个分层的好处不是“绝对最便宜”，而是：
- 昂贵模型只用在方向最值钱的阶段
- 实现阶段避免反复为同一类推理付费
- 轻任务不会污染高价值额度

---

## 个人 / 协作注意

**个人使用**：每次切到旗舰模型、远程 agent 或长自治模式前，先问一句：这一步的错误代价是否真的高到值得用最贵一档？

**团队协作**：
- 建立默认模型层级，不要让每个人按感觉选最贵模型
- 对自动化批任务设置预算上限或任务上限
- 把计划落盘到文件，减少重复规划和重复高价推理
- 定期审查高成本任务，确认它们是否真的带来了对应价值

## 延伸调查方向

1. **Cursor 高阶模式的真实消耗曲线**：前台短对话、Agent、Background Agents 在真实团队里分别多快耗掉额度？
2. **Claude Code 的预算护栏实践**：`--max-budget-usd` 在自动化脚本里应如何设定，才能既稳又不频繁中断？
3. **Aider 的多模型分工收益**：`--weak-model` 与 `--editor-model` 的最佳分工边界在哪里？
4. **Codex CLI 的本地/云端切换策略**：哪些任务值得直接下沉到 `--oss`，哪些任务仍应保留在云端旗舰模型上？
