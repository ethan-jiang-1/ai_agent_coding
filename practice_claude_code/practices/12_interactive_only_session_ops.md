# 12 会话内操作层：旁路动作用 slash command，不要污染主提示词

## 核心习惯

当你的目标不是“让 Claude 产出内容”，而是“控制会话、检查状态、做旁路动作”时，优先用 slash commands，而不是自然语言提示词。

在 Claude Code 上，这已经不是便利入口，而是一层正式控制平面。

## 在 Claude Code 里怎么做

- 想知道当前有哪些内建命令，先用 `/help`。
- 想调权限或切控制策略，优先用 `/permissions`。
- 想看会话或环境状态，先用 `/status` 或相关状态命令。
- 想插入一个不该污染主线程的问题，用 `/btw`。
- 想单独做安全审查，用 `/security-review`。
- 想看后台任务，就用 `/tasks`。
- 如果已经进入定时、轮询、长程自动化，不要还停留在“会话小操作”理解里；`/loop` 和 `/schedule` 应转到第 `13` 章理解。

这条习惯的本质是：

- 产出型请求，走正常提示词
- 会话控制型请求，优先走 slash commands
- 规划、权限、状态、任务观察这些控制动作，不要和实现意图混在一个 prompt 里

## 为什么这很重要

因为很多旁路动作一旦混进主提示词，会带来两个问题：

- 主任务上下文被稀释
- 会话控制动作和实现意图混在一起，后续更难恢复

## 不要这样做

- 不要每次都用自然语言问“现在有哪些命令可用”。
- 不要把“查看状态”“恢复会话”“做安全审查”都写成普通任务提示词。
- 不要把 slash commands 当成实现提示词的装饰语。

## 验收标准

- 你能分清什么时候该发提示词，什么时候该用 slash command。
- 旁路问题不会明显污染主任务上下文。
- 你知道哪些操作只有交互入口，没有稳定外部命令等价物。
- 你知道 `/tasks` 是状态查看层，不等于 durable scheduling。
