# 02 纠错重试：修改原意图，而不是无限追加

## 核心习惯

Claude Code 走偏时，优先回滚文件并用更好的提示词重开，而不是在污染过的上下文里不断追加“还有一个条件”。

## 在 Claude Code 里怎么做

- 文件改坏了，先 `git restore --source=HEAD --worktree <file>`。
- 只是同一任务继续推进，才在原会话里补条件。
- 如果是方向错了，要么新开会话，要么在旧会话上 `--fork-session` 走另一条路径。

真正需要修的不是“AI 说错了一句”，而是：

- 文件状态
- 会话状态
- 提示词本身

## 什么时候可以继续追加

只有在目标没变、只是补一个漏掉的边界时，才适合继续原会话，比如：

- “不要改测试，只改实现”
- “只做第一步，不要扩展”
- “保留现有接口名”

如果已经变成“换方案”，那就不是追加问题，而是重开问题。

## 不要这样做

- 不要把三四轮失败实现都留在一个会话里，最后再指望一条补丁指令救回来。
- 不要把 `--fork-session` 当作自动清理负面 few-shot 的工具。
- 不要用越来越长的提示词去覆盖越来越差的上下文。

## 验收标准

- 如果重试，这次提示词比上一次更明确。
- 如果分叉，旧会话和新方向是可区分的。
- 当前上下文里没有一长串已经被证明错误的实现历史。
