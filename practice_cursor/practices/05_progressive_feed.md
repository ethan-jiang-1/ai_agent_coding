# 05 渐进式信息投喂：先给锚点，再补材料

## 核心习惯

Cursor 虽然有代码库搜索和 `@file` / `@folder` 这类入口，但不代表你应该一开始把所有背景、日志、文档和想法一次灌满。

## 在 Cursor 里怎么做

- 第一轮先给任务目标、关键约束和 2-5 个锚点文件。
- 第二轮再补日志、测试输出、文档、`@Past Chats`。
- 如果关键文件已经开着，优先用 `/Add Open Files to Context` 或 `/Add Active Files to Context`，不要再手工大段粘贴。
- 复杂任务把中间结论外化成文件，而不是一直依赖 tab 记忆。
- 需要精确看某个文件时，用 `@file`；需要让 Cursor 自己扩展检索，再加 `@folder` 或 codebase 搜索。

Cursor 的强项是：

- 你先给锚点
- 它再做检索和截取
- 你按需追加，而不是一次塞满

## 典型节奏

1. 任务目标 + 关键约束
2. `@` 关键文件
3. 补日志 / 测试 / diff
4. 如果任务分步，生成摘要或计划文件
5. 下一步基于文件继续

## 不要这样做

- 不要一开始贴几十段日志和十几个文件内容。
- 不要因为 Cursor 会检索，就完全不给边界。
- 不要把 `/Add Open Files to Context` 当成“一键把所有打开文件都塞进去”的默认动作。
- 不要把“RAG 会自己截取”理解成“精确原文不重要”。

## 验收标准

- 第一轮信息足够清晰，但不是信息大杂烩。
- 后续补充都是按需要增加，而不是一开始塞满。
- 中间关键结论已经部分外化成文件。
