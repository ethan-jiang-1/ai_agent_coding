# 01 会话生命周期：任务边界即 tab 边界

## 核心习惯

在 Cursor 里，最接近“会话边界”的不是某个 CLI session，而是当前 chat tab。任务边界变了，就不要硬续同一个 tab。

## 在 Cursor 里怎么做

- 新任务，直接开新的 chat tab。
- 同一逻辑任务继续推进，才留在原 tab。
- 从只读分析切到实际改代码，如果范围和目标都没变，可以继续；如果模块、风险或目标变了，就开新 tab。
- 跨天回来时，先看旧 tab 是否还干净；不干净就用 handoff 文件或 `@Past Chats` 重开。

一个实用判断：

- 同一任务继续做：留在原 tab
- 任务目标换了：开新 tab
- 旧历史已经开始污染判断：开新 tab

## 什么时候该重开

出现下面这些信号时，不要硬续旧 tab：

- 任务已经从“探索”变成“实施”
- Agent 连续复读你刚否定的方向
- 当前 tab 混入了别的模块和别的方案历史
- 你已经说不清这轮修改到底以哪一版要求为准

## 不要这样做

- 不要把同一个 tab 当成项目总控台。
- 不要因为“历史还在”就默认继续原 tab。
- 不要把 Cursor 的 tab / history 写成 Codex 或 Claude Code 那种明确 `resume` 语义。

## 验收标准

- 当前 tab 只服务一个清楚任务边界。
- 你能解释为什么这次继续原 tab，或者为什么要新开。
- 别人接手时，能看懂这条对话是围绕哪个目标展开的。
