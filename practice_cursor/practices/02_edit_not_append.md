# 02 纠错重试：先回退 agent 改动，再重写提示

## 核心习惯

Cursor 走偏时，优先回到干净文件状态再重试，而不是在一个已经偏航的 tab 里不断追加“还有一个条件”。

## 在 Cursor 里怎么做

- 这次失败主要是 Agent 改坏了文件，先用 checkpoint 回退。
- 混入了手工编辑，或你不确定 checkpoint 能不能完整覆盖时，用 git 回退目标文件。
- 只是漏了一个边界条件，才继续在原 tab 补约束。
- 如果方向已经变了，回退之后直接开新 tab，再用更好的提示词重来。

真正需要修的不是一句话，而是：

- 文件状态
- tab 历史
- 提示词本身

## 什么时候可以继续追加

只有在目标没变、只是补一个漏掉的边界时，才适合继续原 tab，比如：

- “不要改测试，只改实现”
- “只做第一步，不要扩展”
- “只动 `src/auth/`，别碰别的目录”

如果已经变成“换方案”，那就不是追加问题，而是重开问题。

## 不要这样做

- 不要把四五轮失败实现都留在一个 tab 里，最后靠一句补丁提示词救场。
- 不要把 checkpoint 当成 git 的完全替代。
- 不要忘了 checkpoint 只覆盖 agent 更改，不覆盖手工编辑。

## 验收标准

- 如果重试，这次提示词比上一次更明确。
- 如果回退，文件状态已经回到你认可的基线。
- 当前 tab 里没有一长串已经被证明错误的实现历史。
