# Claude 成本优化的习惯导向拆解

这不是对原文的覆盖改写，而是基于原文思路做的一份“习惯导向 + 证据支撑 + Claude Code 操作化”的重构版。

适用场景有两个：

- 手动和 Claude 对话
- 在 Claude Code 里做持续协作

核心观点只有一句话：

**真正省成本，不是少发消息，而是少制造无效上下文，并尽量复用稳定前缀。**

## 为什么用户习惯会影响成本

Anthropic 官方文档说明了几件事：

- 使用量受对话长度、复杂度、已启用功能、所用模型影响
- length limits 的本质是 context window，也就是模型单轮的工作记忆
- Projects 通过 RAG 只加载相关内容，而不是把所有资料全塞进上下文
- tools 和 connectors 本身就是 token-intensive
- prompt caching 只会命中“完全一致”的前缀内容

把这些机制翻译成用户习惯，就是：

- 少发补丁式追问
- 多用固定模板
- 把稳定背景沉淀下来
- 把大材料放进项目
- 根据任务切换模型

---

## 习惯 1：先改原问题，不要连续打补丁

### 你应该怎么做

当回答方向不对，优先编辑原提示词，而不是继续发送：

- 不是这个意思
- 还是不对
- 再换个语气

### 背后的原因

每多一条消息，会话就更长；而会话越长，后续每一轮要重新处理的历史就越多。

### 官方证据

- Anthropic 建议在发送前先检查消息的清晰度和完整性，减少 follow-up messages
- Anthropic 提示工程文档强调：越清楚、越直接，Claude 的结果越准确

### 例子

低效：

```text
给我写一个活动海报文案
太正式了
再活泼点
还要短一点
```

高效：

```text
给我写一个活动海报文案，面向大学生，语气活泼，控制在 80 字以内，突出“免费”和“限时报名”。
```

### Claude Code 里的对应动作

- 把长期约束写进 `CLAUDE.md`
- 不要把长期风格要求反复放在会话里解释

---

## 习惯 2：把相关任务合并成一条消息

### 你应该怎么做

把“总结、提炼要点、给标题、改风格”一次性说完，而不是拆成四轮。

### 背后的原因

Anthropic 官方 usage best practices 明确建议：多个相关任务要尽量合并到一个 message 中。

### 官方证据

- 官方建议：group multiple related tasks or questions in a single message
- 官方提示工程文档强调：明确的任务边界会提升输出准确度

### 例子

```text
请总结这篇文章，提炼 3 个核心观点，给出 2 个可执行建议，并提供 3 个适合公众号发布的标题。
```

### Claude Code 里的对应动作

- 一次说清楚目标、约束、输出格式、验证方式
- 让代理一次规划，而不是来回补充

---

## 习惯 3：长会话要压缩，不要无止境往后拖

### 你应该怎么做

当对话已经很长时，先让 Claude 总结，再开新会话继续。

### 背后的原因

长会话会挤占上下文窗口。真正重要的信息如果没有被提炼出来，就会跟大量历史噪音混在一起。

### 官方证据

- Anthropic 官方说明：对话长度和复杂度会直接影响 usage
- Claude Code 文档说明：`/compact` 可压缩会话
- Claude Code 文档说明：`CLAUDE.md` 在 `/compact` 后会重新加载

### 例子

```text
请把当前对话总结成：
1. 已确认结论
2. 仍未解决的问题
3. 当前约束
4. 建议的下一步
```

### Claude Code 里的对应动作

- 使用 `/compact`
- 把长期规则写进 `CLAUDE.md`，不要依赖“会话记住”

---

## 习惯 4：把稳定背景沉淀为偏好、项目说明和 CLAUDE.md

### 你应该怎么做

如果一条背景信息你说过三次以上，就应该把它移出聊天框。

### 背后的原因

稳定背景是最适合复用的“前缀内容”。如果每次重讲，就等于每次重新付费。

### 官方证据

- Claude 的 Profile preferences 是 account-wide
- Project instructions 只作用于该项目内会话
- `CLAUDE.md` 会在每个 Claude Code session 开始时加载
- 文档建议 `CLAUDE.md` 控制在 200 行以内，越具体越好

### 例子

适合沉淀的内容：

- 默认中文
- 先结论后分析
- 偏好表格还是清单
- 项目测试命令
- 命名规范

### Claude Code 里的对应动作

- `/memory`
- `/init`

---

## 习惯 5：把大文档放进 Projects，而不是重复上传

### 你应该怎么做

合同、长报告、风格指南、PRD、课程资料，这些都应该进入同一个 Project。

### 背后的原因

Projects 不是简单资料夹，而是“检索式上下文入口”。它利用 RAG 只加载相关片段，而不是每轮全量重读。

### 官方证据

- Anthropic 文档明确写到：Projects 使用 RAG，只加载相关内容
- Usage best practices 写到：Project 中的文档会被缓存，后续引用时只有新内容或未缓存内容计入限制

### 例子

如果你一周都在分析同一份行业报告：

- 错误方法：每次开新对话重新上传
- 正确方法：放进一个项目，持续复用

---

## 习惯 6：关闭不用的工具、连接器、搜索与深度思考

### 你应该怎么做

默认只开当前任务真正需要的能力。

### 背后的原因

Anthropic 官方明确说 tools 和 connectors 是 token-intensive。不是功能越多越值，而是功能越准越省。

### 官方证据

- 官方 usage 文档明确指出：工具和连接器很吃 token
- 官方建议：不需要时关闭非关键工具和 connectors

### 例子

- 改文案时不用开搜索
- 做格式转换时不用开 research
- 简单问题先别开重推理

---

## 习惯 7：按任务切模型，而不是一直用最强模型

### 你应该怎么做

建立一个简单分层：

- Haiku 处理轻任务
- Sonnet 处理主力任务
- Opus 处理真正需要高推理的问题

### 背后的原因

模型能力和价格成正相关。任务轻，却一直用高价模型，本质上就是浪费。

### 官方证据

Anthropic 官方价格表显示：

- Haiku 4.5：输入 `$1 / MTok`
- Sonnet 4.5：输入 `$3 / MTok`
- Opus 4.6：输入 `$5 / MTok`

### 例子

- Haiku：翻译、改写、格式整理、提纲
- Sonnet：代码理解、文档重构、主力协作
- Opus：复杂判断、困难拆解、关键决策

### Claude Code 里的对应动作

- `/model`
- `/status`

---

## 习惯 8：用固定模板，主动制造“稳定前缀”

### 你应该怎么做

把高频任务做成模板，只替换变量部分。

### 背后的原因

prompt caching 的命中条件是：前缀必须完全一致。越固定的结构，越接近缓存友好的形态。

### 官方证据

- Anthropic 文档：cache hits require 100% identical prompt segments
- Anthropic 文档：cache stable, reusable content
- Anthropic 文档：place cached content at the prompt's beginning

### 例子

```text
角色：编辑助手
任务：拆解文章
输出：核心观点、证据、反例、改写建议
风格：中文，短句，面向公众号读者
材料：<<<这里替换本次文章>>>
```

---

## 习惯 9：把会话信息和长期记忆分开

### 你应该怎么做

临时信息留在会话里，长期规则写入记忆系统。

### 背后的原因

每个 Claude Code session 都从 fresh context window 开始。如果长期规则不外置，下次还得重建一次。

### 官方证据

- Claude Code 文档：Each session begins with a fresh context window
- `CLAUDE.md` 与 auto memory 会在每次 session 开始时加载

### 例子

临时信息：

- 这次要改哪段文案
- 当前要修哪个 bug

长期信息：

- 这个仓库怎么测试
- 输出应该遵循什么结构

---

## 习惯 10：把省成本理解为“工作编排”

真正有效的不是某一个技巧，而是一整套顺序：

1. 先选模型
2. 再关掉不用的功能
3. 用模板完整发任务
4. 把长期背景移出会话
5. 把大材料放进 Projects
6. 长会话及时压缩
7. 回答方向不对时，优先编辑原消息

这套顺序的本质，是把稳定内容固定下来，把变量内容留给当前任务。
