# 习惯 1：先编辑再追问

## 这个习惯是什么

当 Claude 没理解你的意思时，不要马上追加一串纠错消息，优先编辑原提示词，再重新生成。

## 为什么这会省成本

- 每多一条消息，后续轮次都要带着更多历史继续跑
- 追问式纠错会把真正任务目标埋进噪音里
- 从 KV cache / prefix reuse 的角度看，补丁式对话会让稳定前缀越来越不稳定

## 背后的机制

Anthropic 官方说明，使用量受对话长度和复杂度影响；提示工程文档也强调，表达越直接、越完整，结果越准确。

翻成用户语言就是：

- 能一次讲清，就不要三次补救
- 能改原题，就不要在后面打补丁

## 手动对话例子

低效写法：

```text
帮我写产品介绍
不对，太硬了
也不对，再短一点
还是不对，要面向大学生
```

高效写法：

```text
帮我写产品介绍，面向大学生，语气轻松，控制在 120 字以内，重点突出低门槛和快速上手。
```

## Claude Code 对应动作

- 把长期风格和输出偏好写进 `CLAUDE.md`
- 把项目级规则写进项目记忆，而不是在每个任务里重讲

## 这类习惯真正优化的是什么

它优化的不是“字数”本身，而是减少无效轮次，减少历史污染，让模型更快稳定在正确任务上。
