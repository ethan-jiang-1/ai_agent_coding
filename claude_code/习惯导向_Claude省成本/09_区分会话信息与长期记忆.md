# 习惯 9：区分会话信息与长期记忆

## 这个习惯是什么

把“这次任务临时需要的内容”和“以后每次都需要的规则”分开管理。

## 为什么这会省成本

- 临时信息本来就不应该长期挂在上下文里
- 长期规则如果不沉淀，下次还得重新搭建
- 混在一起会让会话越来越乱

## 背后的机制

Claude Code 文档说明，每个 session 都从 fresh context window 开始；同时 `CLAUDE.md` 和 auto memory 会在每次 session 开始时加载。

这意味着：

- 会话是临时工作台
- 记忆系统才是长期仓库

## 例子

这里也要分清“当前这一轮该说什么”和“以后每轮都不用再说什么”。

适合放在会话里的：

- 这次要修哪个 bug
- 这次文章要面向谁
- 当前任务的特殊约束

适合放在长期记忆里的：

- 仓库测试命令
- 代码风格规范
- 你的默认输出结构

如果一个规则在会话 A、第 1 轮说一次，会话 B、第 1 轮还要再说一次，那它大概率已经不该只留在会话里了。

## Claude Code 对应动作

- 会话里只放这次任务的变量信息
- 长期规则写进 `CLAUDE.md`
- 需要持久化时用 `/memory`

## 这类习惯真正优化的是什么

它优化的是“信息分层”。信息一分层，后续的上下文管理就容易了。
