# 导言：从 Prompt 工程师到上下文工程师

## 你已经会用了，但还没用好

你知道怎么写 Prompt，知道 Cursor 的 Ask / Agent 工作流、Claude Code 的 `plan` permission mode，也知道 Aider 的 `/add` 和 `/ask`，甚至用过 Codex CLI 的沙箱、`resume` / `fork` 或 `exec`。你用 AI 写代码已经有一段时间了，效率确实提升了。

但你也遇到过这些情况：

- 同一个 AI，早上用很顺，下午越来越蠢，开始重复已经排除的方案
- 一个"简单"的重构任务，AI 修了十几个来回，最后代码比原来还乱
- 月底看账单，Token 消耗比预期高出好几倍，但不知道钱花在哪里
- 团队里每个人的 `.cursor/rules`、`CLAUDE.md` 或 `AGENTS.md` 都不一样，AI 给不同人的输出风格完全不一致

这些问题的根源，**几乎都不是模型不够好，而是上下文管理出了问题。**

---

## 范式转移：上下文窗口不是聊天记录

早期用 ChatGPT 写代码，上下文就是聊天历史——你问，它答，偶尔粘贴一段代码。这个心智模型在 2023 年是够用的。

但现在的 AI Coding 工具已经完全不同了。Claude Code 可以自主读取几十个文件、执行终端命令、调用 MCP 服务器；Cursor 的 Agent 模式可以跨文件修改并自动运行测试；Aider 会用 `repo map` 压缩代码库轮廓；Codex CLI 既能在沙箱里执行命令，也支持持续会话、非交互式执行和子智能体能力。

**这时候，上下文窗口不再是聊天记录，而是一块高度易失、预算严格受限的计算内存。**

类比一下：
- 上下文窗口 = RAM（有限，断电即失）
- 代码库 = 硬盘（持久，可随时读取）
- 规则文件（如 `CLAUDE.md` / `AGENTS.md` / `.cursor/rules`）= 操作系统启动配置（每次启动自动加载）
- 会话历史 = 进程内存（进程结束即清空）

低效用户把 RAM 当硬盘用——把所有东西都塞进去，然后抱怨"内存不够"。高效用户像操作系统一样管理内存——按需加载，及时释放，关键状态持久化到磁盘。

## 成本的真实构成

在 AI Coding 语境下，"成本"不只是 API 账单，而是五个维度：

| 成本维度 | 具体表现 | 常被忽视的原因 |
|---------|---------|--------------|
| Token 消耗 | 输入 + 输出 + 推理 Token | 推理 Token 按输出价格计费，可造成 2-50x 倍增 |
| API 费用 | BYOK 模式下直接转化为财务支出 | 工具调用死循环时指数级增长 |
| 套餐额度 | 订阅制产品的高级模型请求次数 | 高峰时段有隐性乘数，很多用户不知道 |
| 等待时间 | 模型排队、长上下文处理延迟 | 时间成本往往比 Token 成本更贵 |
| 返工成本 | 上下文污染导致的代码回滚和二次验证 | 行业数据：每位开发者每年损失 $4,500-$8,800 等值工时 |

**最贵的成本往往是最后一项：返工。** 而返工的根源，几乎都是上下文管理失当。

## 本文讲什么，不讲什么

**讲**：
- 已经在用 AI Coding 工具的开发者，如何调整工作习惯来最大化 AI 能力
- 每个习惯背后的原理（为什么这样做有效，不只是"最佳实践"）
- 跨工具通用的做法（不绑定某一个产品）
- 可以继续深挖的真实场景和数据

**不讲**：
- 如何安装和配置 AI Coding 工具（你已经会了）
- 哪个工具最好（取决于场景，没有唯一答案）
- 提示词技巧大全（那是另一个话题）
