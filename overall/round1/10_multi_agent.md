# 第十章：多智能体分流——Writer / Reviewer 双路并发

## 习惯

**大型重构时，用两个独立上下文窗口分别承担"编写者"和"审查者"角色。审查者的干净上下文——没有编写者的试错历史——是发现隐性错误的关键。**

---

## 正确的做法

### 为什么需要双路架构

单个 AI 会话在大型任务中面临一个根本性的局限：它既是执行者，又是审查者，但它"看过"所有的中间过程，形成了认知偏差。

**核心问题**：一个在会话中经历了"先尝试 A 方案，失败；再尝试 B 方案，调整；最终完成 C 方案"的 AI，在审查自己生成的代码时，会潜意识地"理解"那些只有在上下文历史中才能看懂的决策。而一个看到同样代码的新鲜眼睛，可能会立刻发现"这里为什么不用标准的方式实现？"

审查者需要的，正是这双"新鲜眼睛"。

### 双路并发的基本模式

```
Session A（编写者）                Session B（审查者）
─────────────────────            ─────────────────────
上下文：                          上下文：
  - 项目规则（CLAUDE.md）            - 项目规则（CLAUDE.md）
  - 任务描述                         - 只有 diff 文件
  - 探索历史
  - 试错记录
  - 最终实现代码

→ 生成 diff 文件 ──────────────→ 审查 diff，无任何历史包袱
                                  发现：
                                  - 命名不一致
                                  - 遗漏的边界条件
                                  - 架构模式偏离
                                  - 潜在的安全问题
```

### 各工具的实现方式

**Claude Code（CLI）——Git Worktree 物理隔离**

```bash
# 为大型重构创建独立的工作树
git worktree add .worktrees/oauth2-migration -b feature/oauth2-migration

# Session A：在 worktree 里进行实现
cd .worktrees/oauth2-migration
claude "按照 .claude/plans/oauth2-migration.md 的计划，实现 OAuth2 迁移"

# 实现完成后，生成 diff
git diff main > /tmp/oauth2-migration.diff

# Session B（全新终端，干净上下文）：审查
claude "审查这个 diff，检查：
1. 是否有逻辑漏洞或边界条件遗漏？
2. 是否与现有代码的架构模式一致？
3. 是否有潜在的安全问题？
4. 是否有可以简化的过度工程化部分？
不要给出修改建议，只列出问题清单。" < /tmp/oauth2-migration.diff
```

**Cursor——独立 Composer 窗口**

```
# Composer 1（编写者）
Shift+Tab → Plan Mode
→ 生成实施计划
→ 切换回执行模式，按计划实现

# Composer 2（审查者）——新建一个独立的 Composer
"以下是一段代码变更的 diff。
你没有这段代码的任何背景信息。
作为一个严格的代码审查者，指出所有你认为有问题的地方：
[粘贴 git diff 输出]"
```

**子智能体任务委派**

对于大型代码库的探索任务，不要在主会话里执行：

```bash
# 错误的做法（主会话读取大量文件，污染上下文）
claude "读取所有 auth 相关的文件，理解现有认证架构"
# 这会把几十个文件塞进主会话的上下文

# 正确的做法（委派给子智能体）
claude "用子智能体探索 src/auth/ 目录，
理解现有的认证流程，
只把架构摘要（不超过 500 字）返回给我"
# 子智能体在独立上下文中工作，主会话只接收摘要
```

### 并行子任务的分流

对于可以并行处理的子任务，同时开启多个实例：

**Codex CLI——高并发子智能体（专为并行设计）**
```bash
# Codex CLI 原生支持在沙箱内同时运行多个子智能体
# 每个子智能体在独立的沙箱上下文中运行，互不干扰

codex "并行执行以下三个任务：
  1. 分析 src/auth/ 的现有实现，生成架构摘要
  2. 分析 tests/auth/ 的现有测试，找出缺失的测试用例
  3. 检查 src/config/ 里与 auth 相关的配置项"

# 三个子智能体同时工作：
# - 各自在独立的沙箱上下文中读取文件（不共享状态）
# - 各自完成后，Codex CLI 汇总结果返回给主会话
# - 主会话上下文只增加三份摘要，而非三次完整的文件探索历史
```

**适合 Codex CLI 并行的场景**：
- 大批量代码审查（同时分析多个模块的安全问题）
- 并行探索陌生代码库的不同子系统
- 同时运行多组测试并验证结果

**Claude Code——多终端手动并行**
```bash
# 终端 1：实现认证中间件
claude "实现 src/auth/middleware.ts 的 OAuth2 验证逻辑"

# 终端 2（同时）：实现配置层
claude "实现 src/config/oauth2.config.ts 的配置结构"

# 终端 3（同时）：更新测试文件
claude "为 OAuth2 功能更新 tests/auth/ 下的测试用例"

# 各自完成后，人工整合并做最终审查
```

**注意**：并行任务需要确保各自的范围不重叠，避免多个 AI 同时修改同一个文件导致冲突。

## 错误的可能

### 错误一：期待同一个 AI 会话有效地自我审查

```
# 无效的自我审查
用户：审查一下你刚才写的代码
AI：这段代码看起来不错，[解释为什么每个决策都是合理的]
```

这不是 AI 在撒谎，而是认知偏差的必然结果。AI 在写完这段代码后，"理解"了所有的中间决策，它很难站在一个纯粹的外部视角来审查。

**类比**：要求一个刚写完代码的工程师立刻审查自己的代码，发现的问题数量会远少于休息一晚上后再审查，更少于另一个工程师来审查。

### 错误二：Trae 等工具的并行任务滥用

Trae 和 Codex macOS 应用提供了强大的并发任务功能，允许同时运行 10-15 个智能体。这是一把双刃剑。

**错误场景**：
```
用户：同时处理以下 15 个任务：
1. 重构 auth 模块
2. 优化数据库查询
3. 更新 API 文档
... (15 个任务)
```

**真实后果**：如果其中任何一个智能体的方向错了，15 个智能体可能同时在云端陷入重试循环，不仅产生巨额的上下文污染，还会瞬间榨干当月的 Fast Queue 额度。有用户报告在几小时内就耗尽了整月的配额。

**正确的并行模式**：
- 并行任务的范围必须完全独立（不共享文件）
- 每个任务必须有明确的停止条件
- 任务数量不超过你能同时监控的数量

### 错误三：跨智能体上下文污染

在多智能体协同工作流中，如果不同角色的智能体共享了同一个记忆池：

```
错误场景：
- Agent A（编写者）产生了一个错误的架构假设
- 这个假设被写入了共享记忆库
- Agent B（审查者）读取了共享记忆库
- Agent B 的审查基于了 Agent A 的错误假设
- 整个审查过程变得无效
```

这是多智能体系统最常见的失败模式之一。审查者必须在**隔离的**上下文中工作，只看 diff，不看历史记录。

## 背后的原理

### 认知偏差的不可避免性

心理学研究表明，人类在审查自己刚完成的工作时，会产生"确认偏差"——潜意识地寻找支持自己决策的证据，而忽视反驳的证据。

LLM 在这方面更严重：它的上下文里包含了所有的中间决策过程，它"知道"为什么每一行代码都是这样写的。这让它在审查时很难像一个"第一次看到这段代码的人"那样思考。

**扇出/扇入（Fan-out / Fan-in）模式**：借用分布式计算的概念，大型任务被分解为多个独立的子任务（扇出），各子任务在隔离环境中并行执行，最终结果汇总合并（扇入）。物理隔离的上下文确保了每个智能体的判断不受其他智能体的干扰。

### Git Worktree 的隔离机制

Git Worktree 创建了代码库的独立工作副本：

```bash
git worktree add .worktrees/feature-x -b feature-x
```

这意味着：
- `.worktrees/feature-x` 和主目录是同一个 Git 仓库的不同工作目录
- 在 worktree 里的修改不会影响主目录
- 可以随时切换回主目录，worktree 里的修改独立保留
- 审查完成，合并前可以先对比 `git diff main..feature-x`

Worktree 的物理隔离，配合独立的 Claude Code 实例，实现了完美的"编写者"和"审查者"分离。

### 子智能体的上下文经济学

在主会话中探索大型代码库的代价极高：
- 读取 50 个文件 → 主会话上下文膨胀数万 Token
- 这些探索记录永久留在上下文里，影响后续所有交互

委派给子智能体的代价：
- 子智能体在独立上下文中读取 50 个文件
- 生成一个 500 Token 的架构摘要返回给主会话
- 主会话只增加了 500 Token，而不是数万 Token

**Token 节省比例**：通常可以减少 80-95% 的探索相关 Token 消耗。

## 延伸的习惯

### 标准化审查提示词

```markdown
# reviewer-prompt.md（可复用的审查提示词模板）

你是一个严格的代码审查者。
你没有这段代码的任何背景信息——这是你第一次看到它。

请检查以下方面：
1. 逻辑正确性：是否有边界条件未处理？
2. 架构一致性：是否与常见的代码模式保持一致？
3. 安全性：是否有 SQL 注入、XSS、认证绕过等风险？
4. 性能：是否有明显的性能问题？
5. 过度工程化：是否有不必要的复杂性？

输出格式：
- [CRITICAL] 必须修复
- [HIGH] 强烈建议修复
- [MED] 建议修复
- [LOW] 可选优化

只列问题，不要给出修复方案。
```

### 轻量版：让 AI 假装是新人

如果不想开启独立会话，可以通过提示词模拟"新鲜视角"：

```
"忘掉你刚才写的所有代码。
假设你是一个新加入团队的工程师，
第一次看到以下代码。
你会有哪些疑问或担忧？
[粘贴代码]"
```

这不如真正的隔离上下文，但比直接让 AI "审查你刚才写的代码"要好得多。

## 个人 / 协作注意

**个人使用**：对于小任务（单文件修改、简单的 Bug 修复），不需要双路架构，直接在主会话完成即可。双路架构适用于影响超过 3 个文件或引入新模块的任务。

**团队协作**：
- 把 "AI 生成代码 → AI 审查 diff → 人工最终确认" 建立为团队的标准代码审查流程
- Git Worktree 配合 PR 工作流：在 worktree 里完成开发，合并到主分支前必须有独立 AI 审查
- 并行任务需要明确的任务分配：哪个人负责监控哪个 Agent，避免所有人都以为别人在监控
- 定期审查多智能体工作流的成本，防止并行任务滥用导致额度耗尽

---

## 延伸调查方向

1. **双路审查 vs 单路自我审查的 Bug 发现率**：在相同的代码变更上，让同一个 AI 会话自我审查 vs 用独立上下文的 AI 审查，发现的问题数量和严重性有多大差异？有没有可复现的实验？

2. **子智能体委派的实际 Token 节省**：在真实项目中，委派探索任务给子智能体（只返回摘要）vs 在主会话直接探索，Token 消耗差异有多大？摘要的信息损失率有多高？

3. **Git Worktree 在 AI 工作流中的采用情况**：有多少团队在日常 AI Coding 工作流中使用 Git Worktree 进行隔离？Worktree 引入的复杂性是否值得？

4. **Trae 并行任务的额度消耗模式**：在不同配置下（5 个并行 vs 10 个并行），Trae 的 Fast Queue 额度消耗速度有多大差异？有没有关于最优并行度的实测数据？
