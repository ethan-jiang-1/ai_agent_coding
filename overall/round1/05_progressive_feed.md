# 第五章：渐进式信息投喂——用指针代替粘贴

## 习惯

**能让工具自己读文件，就不要手工搬运全文。把“位置”和“目的”说清楚，通常比把一整篇文档塞进上下文更有效。**

---

## 正确的做法

### 核心原则：先给入口，再给内容

把信息投喂给 AI，大致有两种方式：
- **内容投喂**：把文档全文复制粘贴进对话
- **指针投喂**：告诉 AI 去哪个文件、目录或官方来源读取信息

对能读工作区、能跑命令、能接 MCP 的工具来说，第二种通常更稳。你要做的是把范围框清楚，而不是替它搬运材料。

### 已审计工具里的安全写法

**Cursor**
```text
"参考 @src/auth/login.ts 的实现方式，为 register.ts 写类似的逻辑"
"根据 @docs/api-spec.md 中的 /users 契约，补全客户端调用"
```

当前这轮最稳的写法是：把 Cursor 描述为可以把工作区文件和规则上下文挂入对话；更细的 UI 按钮、快捷键和版本差异，不要写死。

**Claude Code**
```bash
# 通过管道传入错误日志
cat error.log | claude "分析这个错误，找出根本原因"

# 在提示词里明确让它先读哪些文件
claude "先阅读 docs/auth-flow.md 和 src/auth/ 目录，再给我最小改动方案"
```

如果需要外部系统，优先走 Claude Code 的正式 MCP 入口，而不是把网页教程全文贴进对话。

**Aider**
```bash
# --read 放稳定参考材料
aider --read docs/api-spec.md --read CONVENTIONS.md src/auth/login.ts

# 只把会改的文件显式加进来
/add src/auth/login.ts

# 日志直接走管道
make test 2>&1 | aider "修复这些测试失败"
```

`--read` 和 `/read` 的价值不只是“只读”，也是把参考材料和可编辑范围分层。

**Codex CLI**
```bash
# 错误日志和测试输出直接走管道
cat error.log | codex "分析根本原因"
make test 2>&1 | codex "修复这些测试失败"

# 在提示词里明确要读的文件
codex "先阅读 docs/auth-flow.md 和 src/auth/login.ts，再给出修改方案"

# 只读分析也可以走非交互式执行
codex exec --sandbox read-only "先阅读 docs/api-spec.md，只输出实施计划"
```

Codex CLI 这一轮能安全写成事实的是：它依赖提示词描述、工作区读取和正式 MCP 机制，不要把它写成支持 Cursor 那套 `@` 语法。

### 外部文档的处理顺序

优先级建议：
1. 官方机器可读来源，如 `llms.txt`、OpenAPI、官方 MCP
2. 项目内已经落盘的规范文件
3. 官方网页的精确页面和精确段落
4. 搜索结果

如果只有 HTML 页面，不要整页复制。给 AI 一个准确入口，说明“看哪一页、哪一节、为了什么看”。

### 渐进式披露的节奏

不是所有信息都该第一轮注入：

```text
第一轮：任务描述 + 核心约束
第二轮：按需补具体文件或文档
第三轮：补测试、日志或 diff
```

原则很简单：在 AI 确实需要某段材料之前，不要抢先把一切都塞进去。

## 错误的可能

### 错误一：暴力粘贴整篇 API 文档

```text
# 错误做法
[粘贴 200 行第三方 API 文档]
"根据上面的文档，帮我实现支付接口"
```

问题不只是一口气多花了 Token。更大的风险是：
- 里面真正相关的内容可能只占很小一部分
- 页面文本常常夹带导航、广告和格式噪音
- 文档版本可能已经过时

对 API 集成这类时效性高的任务，更稳的做法是让工具读取官方契约，而不是让模型自己猜教程版本。

### 错误二：一次性挂载过多无关文件

```text
# 错误的做法
@src/auth/login.ts
@src/auth/register.ts
@src/auth/middleware.ts
@src/models/User.ts
@src/models/Session.ts
@src/config/database.ts
"帮我修改密码重置功能"

# 更好的做法
@src/auth/reset-password.ts
@src/models/User.ts
"帮我修改密码重置功能，确保 token 有 15 分钟过期时间"
```

挂载越多无关文件，注意力越分散。知道问题范围时，就要精确缩小范围。

### 错误三：把“Web Search”当成最新 API 契约

```text
# 危险的做法
"搜索一下 Stripe 的最新 API，然后帮我实现支付"
```

搜索适合找入口，不适合替代契约源。真正要生成调用代码时，应尽量回到官方 API 文档、OpenAPI 或官方 MCP。

### 错误四：手动复制错误日志

```bash
# 更稳的做法
make test 2>&1 | claude "修复这些测试失败"
cat error.log | codex "分析根本原因"
```

管道传入的关键价值是完整性。手工复制很容易丢掉堆栈尾部、颜色去除后的上下文，或把多段输出打乱。

## 背后的原理

### 注意力是稀缺预算

上下文里的无关材料不是“中性背景”，而是在主动消耗模型的注意力预算。把一大堆和当前问题无关的内容塞进来，通常只会让真正相关的信息更难被抓住。

### 文档有时效性

代码库里的本地文件、官方机器可读规范、官方 MCP，通常比“搜索到的一篇教程”更接近当前真实接口。对快速变化的 API 和 SaaS 产品尤其如此。

### 管道保证原始性

日志、测试失败、命令输出，本来就是终端产物。能原样通过管道喂给工具，就不要先经过人工删改再贴进去。

## 延伸的习惯

### 建立项目文档索引

可以在 `CLAUDE.md`、`AGENTS.md` 或团队约定文档里维护一个简短索引：

```markdown
## 文档索引
- API 规范：`docs/api-spec.md`
- 数据库 Schema：`docs/schema.md`
- 架构决策记录：`docs/adr/`
- 第三方服务集成：`docs/integrations/`
```

这样 AI 需要补材料时，先有一个稳定入口，而不是每次靠人工回忆。

### MCP 按场景连接

MCP 更适合“按需挂载”，而不是永久常驻：

```text
调试阶段：连接错误监控或工单系统
API 集成阶段：连接官方契约或 OpenAPI
任务结束：收回与当前问题无关的外部入口
```

### 长日志先做轻量预处理

对于特别长的输出，可以先缩小范围再喂给工具：

```bash
tail -50 error.log | claude "分析根本原因"
grep -v "DeprecationWarning" error.log | codex "总结真正的失败原因"
grep "ERROR\\|FATAL" app.log | aider "找出共同根因"
```

## 个人 / 协作注意

**个人使用**：养成“先给路径、再给内容”的习惯，通常会减少 Token 消耗，也会减少因为贴错版本文档造成的返工。

**团队协作**：
- 团队共享的 MCP 配置，应该通过各工具的正式配置机制纳入版本控制
- Web Search 更适合找入口，不适合替代团队内部规范和官方契约
- 对常用第三方 API，优先沉淀成固定入口，而不是每个人都临时搜索

---

## 延伸调查方向

1. **引用 vs 粘贴的 Token 效率差异**：在同一任务里，引用本地文件和直接粘贴全文，实际输出质量和成本差异有多大？
2. **llms.txt 的实际覆盖率**：主流框架和 SaaS 文档里，`llms.txt` 的采用率和质量到底如何？
3. **MCP 在 API 集成场景的收益**：用官方契约源替代网页搜索后，首轮可运行代码的成功率能提高多少？
4. **日志完整性对诊断质量的影响**：管道传入与手工复制相比，AI 诊断准确率差异有多大？
