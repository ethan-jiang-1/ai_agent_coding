# 第七章：模型路由分级——按任务复杂度选择正确的模型

## 习惯

**架构规划用旗舰模型，日常代码实现用中档模型，文件探测与子智能体任务用轻量模型。不同复杂度的任务，坚决不一视同仁。**

---

## 正确的做法

### 三级路由框架

| 任务类型 | 典型场景 | 推荐模型层 | 理由 |
|---------|---------|-----------|------|
| **重任务** | 跨模块重构、架构设计、疑难 Bug 排查 | Claude Opus / GPT-5 级旗舰模型 | 需要深层推理，方向错误的代价极高 |
| **中任务** | 常规功能实现、单元测试编写、代码审查 | Sonnet / GPT-4.x / GPT-4o 一类中档模型 | 覆盖大多数日常任务，性价比最优 |
| **轻任务** | 文件检索、日志分析、格式转换、Linting 修复、探索子任务 | Haiku / Flash / 本地模型 | 速度快、成本低，这类任务通常不需要深层推理 |

### 各工具的模型切换操作

**Claude Code（CLI）**
```bash
# 重任务：显式切到旗舰模型
claude --model opus "先做架构方案，再给我实施计划"

# 日常实现：默认使用中档模型
claude --model sonnet "按已确认计划实现这个功能"
```

**注意**：不要在正文里虚构某个“固定的子智能体模型配置键”。Claude Code 当前官方可直接确认的是 `--model`，以及 subagents 的存在；至于子智能体是否单独绑定模型，应以当期官方文档为准。

**Cursor**
```
# 日常使用：保持 Auto，或手动选中档模型
# 重任务：显式切到旗舰模型
# 轻任务：Ask + 更轻量模型
# 具体计费和可见模型以 Cursor 当前官方 pricing / model chooser 为准
```

**Aider（CLI）**
```bash
# 重任务：使用 Opus
aider --model claude-opus-4-6 src/core/

# 中任务：使用 Sonnet（默认推荐）
aider --model claude-sonnet-4-6 src/features/

# 轻任务：使用 Haiku 或本地模型
aider --model claude-haiku-4-5 src/utils/

# 还可以把弱模型和编辑模型分开
aider --model claude-sonnet-4-6 \
      --weak-model claude-haiku-4-5 \
      --editor-model claude-haiku-4-5 src/utils/
```

**Codex CLI**
```bash
# 重任务：显式用旗舰模型
codex --model gpt-5 "先做架构诊断，再给出重构路径"

# 一次性执行任务：也可以在 exec 里显式设模型
codex exec --model gpt-5 --ephemeral "按计划修改 src/auth/"

# 轻任务：如果只是本地探索、格式转换或快速辅助，
# 可以考虑更轻的模型或本地 provider（以当前环境支持为准）
codex --oss "解释这个错误日志"
```

**本地模型的使用场景**

通过 Ollama 运行本地模型，用于轻量任务：
```bash
# 安装并运行 Qwen2.5-Coder（7B 参数，轻量编码模型）
ollama run qwen2.5-coder:7b

# 在 Aider 或 Continue 中配置使用本地模型
aider --model ollama/qwen2.5-coder:7b
```

适合本地模型的场景：
- 代码格式化建议
- 简单的变量重命名
- 日志文件分析
- 生成模板代码

### 推荐的混合工作流

```
阶段 1：架构规划（旗舰模型）
  在 Web 端（claude.ai）或通过 API 使用 Opus
  输出：详细的实施计划文档
  ↓ 把计划存入文件（.claude/plans/feature-x.md）

阶段 2：核心实现（中档模型）
  在 Claude Code 或 Cursor 中使用 Sonnet
  输入：阶段 1 生成的计划文档
  输出：核心业务逻辑代码
  ↓ 提交代码

阶段 3：样板代码生成（轻量模型）
  使用 Haiku 或本地模型
  生成：单元测试模板、类型定义、配置文件
  ↓ 人工审查后提交
```

## 错误的可能

### 错误一：无论任务大小，长期默认使用旗舰模型

最普遍也最贵的错误。用 Opus 或 GPT-5 来：
- 修改变量名
- 格式化代码
- 生成 README
- 写简单的 CRUD 接口

**真实后果**：旗舰模型通常更贵，也更容易更快吃掉订阅额度或 API 预算。把它长期用于重命名、格式化、README 这类任务，往往是最先失控的成本项。

安全写法是：
- 复杂规划、跨模块重构、方向不明确的问题，才升级到旗舰模型
- 常规实现默认留在中档模型
- 检索、格式转换、样板生成尽量下沉到轻量模型或本地模型

### 错误二：不理解推理 Token 的计费方式

更准确的说法是：更强的推理模式通常会带来更高延迟和更多 token 消耗，但具体计费方式要看当期 provider 的官方说明，不能把某一家当前的计费细节硬写成普遍规律。

**错误场景**：让高推理模式处理“改变量名”“改文案”“补注释”这类简单任务。即使结果正确，推理开销通常也不值得。

### 错误三：混淆"旗舰模型"和"正确使用"

很多人认为用了旗舰模型就能得到更好的结果。实际上，如果上下文管理很差（大杂烩会话、规则文件臃肿），即使是最强的模型也会表现糟糕。

**正确的优先级**：
1. 先把上下文管理做好（前六章的习惯）
2. 在干净的上下文下，选择合适的模型
3. 不需要旗舰模型的任务，坚决不用

一个用 Sonnet 在干净上下文里工作，通常比用 Opus 在混乱上下文里工作效果更好，成本更低。

## 背后的原理

### 模型能力与任务需求的匹配

旗舰模型的核心优势在于**深层推理能力**：
- 理解复杂的多步骤依赖关系
- 在多种架构方案之间权衡
- 处理模糊需求时的自我纠正能力

但这些能力对于大多数日常编码任务来说是**溢出的**：
- 写一个工具函数：不需要深层推理
- 格式化代码：不需要任何推理
- 生成样板测试：模式匹配，不需要推理

**类比**：请一个顶尖外科医生来处理普通的擦伤，技术上完全没问题，但成本结构完全不合理。合理的做法是：普通擦伤 → 护士处理；复杂手术 → 专科医生。

### 模型路由的经济学

假设一个开发者每天的工作分布：
- 重任务（架构设计、复杂 Bug）：10%
- 中任务（功能开发、代码审查）：60%
- 轻任务（文件检索、格式化、测试模板）：30%

如果全部使用旗舰模型，成本一定向最贵的一档靠拢；如果按任务分层，通常能显著降低预算压力，同时把最贵的模型留给真正需要它的场景。

## 延伸的习惯

### 在 Aider 中实现两阶段工作流

```bash
# 阶段 1：用 Opus 规划（只读模式，不修改文件）
aider --model claude-opus-4-6 \
      --message "分析 src/auth/ 的现有实现，
                 给我一个 OAuth2 迁移计划，不要修改任何文件"

# 计划生成后，保存到文件
# 阶段 2：用 Sonnet 执行（按计划修改文件）
aider --model claude-sonnet-4-6 \
      --read .claude/plans/oauth2-migration.md \
      src/auth/
```

### OpenRouter / 本地 provider 接入低成本模型

对于不需要高质量输出的轻任务，可以通过 OpenRouter 接入性价比更高的模型：

```bash
# 在 Aider 中用 OpenRouter 接入低成本模型
aider --model openrouter/deepseek/deepseek-coder \
      src/utils/

# 低成本模型适合轻任务，不适合直接接手关键架构决策
```

### 为团队建立模型路由规范

```markdown
# 团队 AI 工具使用规范

## 模型选择原则
- 需求分析、架构讨论：Opus（每次使用前在团队频道通知）
- 日常功能开发：Sonnet（默认选择）
- 代码格式化、测试生成、文档：Haiku 或本地模型

## 旗舰模型使用审批
以下情况使用 Opus 无需审批：
- 跨模块重构（影响超过 5 个文件）
- 系统级 Bug 排查（生产问题）
- 新功能的初始架构设计

以下情况不应使用 Opus：
- 单文件修改
- 格式化、重命名
- 生成 README 或注释
```

## 个人 / 协作注意

**个人使用**：建立"默认 Sonnet，显式升级"的习惯。当你想切换到旗舰模型时，问自己：这个任务真的需要深层推理吗？如果答案是"也许不需要"，先用 Sonnet 试一次。

**团队协作**：
- 建立模型使用规范，避免个人习惯差异导致的成本失控
- 如果使用共享 API Key，旗舰模型的使用应该有记录（哪个任务用了 Opus，为什么）
- 定期审查 API 账单，分析高成本使用是否对应高价值任务

## 延伸调查方向

1. **Sonnet vs Opus 在日常编码任务上的质量差距**：在不需要深层推理的任务（CRUD 开发、测试生成、格式化）上，Sonnet 和 Opus 的输出质量差距有多大？有没有系统性的基准测试？

2. **本地模型的实际可用性边界**：Qwen2.5-Coder 7B 或 DeepSeek-Coder 6.7B 在哪些类型的任务上可以替代云端 API？量化准确率和速度的权衡？

3. **工具内模型分工的真实边界**：Claude Code 的 subagents、Aider 的 `weak-model` / `editor-model`、Codex 的主模型选择，在真实项目里分别适合承担什么工作？哪些是官方支持，哪些只是用户习惯？

4. **推理 Token 的消耗分布**：在启用 extended thinking 模式的 Claude 中，推理 Token 占总 Token 消耗的比例在不同任务类型下是多少？何时推理 Token 的额外成本是值得的？
