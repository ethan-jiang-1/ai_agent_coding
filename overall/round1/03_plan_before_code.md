# 第三章：先规划，后执行——探索与执行强制分离

## 习惯

**第一轮交互只允许 AI 探索和规划，明确禁止直接写代码。审查计划通过后，才进入执行阶段。**

---

## 正确的做法

### 四步走：Explore → Plan → Implement → Commit

这不是建议，而是强制流程。每一步都有明确的产出和验收标准：

| 阶段 | AI 被允许做什么 | AI 被禁止做什么 | 人类的动作 |
|------|----------------|----------------|-----------|
| Explore | 读取文件、搜索代码、提问澄清 | 修改任何文件 | 回答 AI 的问题 |
| Plan | 输出实施计划（文件路径 + 变更大纲） | 修改任何文件 | 审查并批准计划 |
| Implement | 按计划修改代码 | 超出计划范围 | 监控执行，随时叫停 |
| Commit | 提交代码 | 修改提交信息之外的内容 | 最终确认 |

### 各工具的具体操作

**Claude Code（CLI）**
```bash
# 正式做法：直接用 plan permission mode
claude --permission-mode plan "先不要写代码。探索 src/auth/ 目录，理解现有的登录流程，
然后给我一个将 JWT 认证迁移到 OAuth2 的实施计划，
包括需要修改的具体文件和每个文件的变更内容。"
```

**Cursor**
```
# 优先用 Ask 做只读探索
# 如果团队配置了自定义 Plan mode，再用它承接规划
"[ASK / PLAN ONLY - DO NOT WRITE CODE]
分析 src/components/UserDashboard.tsx 的现有结构，
列出将其拆分为三个子组件的具体方案，
包括每个子组件的 props 接口定义。"
```

**Aider（CLI）**
```bash
# Aider 没有官方 Plan Mode，但有官方只读入口
aider --read src/auth/login.ts --read src/auth/middleware.ts
aider --message "不要修改任何文件。基于已读文件给我一个重构计划，我确认后再执行。"
```

**Codex CLI**
```bash
# 用只读 sandbox 做规划阶段
codex exec --ephemeral -s read-only "不要修改任何文件。分析 src/auth/ 的现有实现，
列出将 JWT 迁移到 OAuth2 需要修改的文件清单
和每个文件的变更摘要。只输出计划，不执行。"

# 计划确认后，再运行执行命令
codex exec --ephemeral -s workspace-write "按照以下计划修改代码：[粘贴上一步的计划]"
```

### 计划的验收标准

一个合格的实施计划应该包含：

```markdown
## 实施计划：JWT → OAuth2 迁移

### 受影响的文件
1. `src/auth/middleware.ts` - 修改 token 验证逻辑
2. `src/auth/login.ts` - 添加 OAuth2 授权码流程
3. `src/config/auth.config.ts` - 新增 OAuth2 配置项
4. `tests/auth/login.test.ts` - 更新测试用例

### 不会修改的文件
- `src/models/User.ts`（用户模型不变）
- `src/api/routes.ts`（路由不变）

### 执行顺序
1. 先修改配置文件（无依赖）
2. 修改中间件（依赖配置）
3. 修改登录逻辑（依赖中间件）
4. 更新测试（最后）

### 潜在风险
- 现有的 JWT token 在迁移期间会失效，需要处理过渡期
```

**如果 AI 给出的计划没有具体文件路径，要求它重新规划。** 模糊的计划是执行失败的前兆。

## 错误的可能

### 错误一：直接让 AI 开始写代码

最常见的错误：

```
# 错误的做法
"帮我把整个老旧的 MVC 架构迁移至最新的前后端分离模式"

# AI 开始修改文件...
# 修改到第 5 个文件时，忘记了第 1 个文件里设定的类型定义
# 整个项目无法编译
```

**真实后果**：在没有规划阶段的情况下，AI 在处理大型重构时，通常在修改到第 5-7 个文件时开始出现逻辑不一致。原因是上下文窗口有限，早期的决策细节会被后来的内容"挤出"注意力范围。

这不是模型的问题，而是工作流的问题。同样的任务，加上规划阶段，成功率会显著提升。

### 错误二：接受模糊的计划就开始执行

```
# 不合格的计划（不要接受）
"我会修改认证相关的文件，添加 OAuth2 支持，
并更新相关的测试。"

# 合格的计划（才能接受）
"我会修改以下 4 个文件：
1. src/auth/middleware.ts（第 23-45 行，替换 JWT 验证逻辑）
2. ..."
```

接受模糊计划的后果：AI 在执行时会"自由发挥"，修改你没有预期的文件，或者遗漏关键的边界情况。

### 错误三：在执行阶段临时扩大范围

```
# 执行阶段的错误追加
"顺便把这个模块的代码风格也统一一下"
"既然你在改这里，把旁边的那个函数也重构了吧"
```

**真实后果**：这会让 AI 在执行阶段进入"探索模式"，开始读取更多文件，上下文迅速膨胀，原有的计划约束被稀释。

## 背后的原理

### AI 是"热情的初级工程师"

当前阶段的代码智能体，更像是一名充满热情但缺乏业务背景的初级工程师。他们执行力很强，但如果没有明确的边界约束，会倾向于：
- 过度工程化（把一个简单的修改变成一个复杂的重构）
- 引入不必要的新依赖
- 修改"顺手"但超出范围的代码

规划阶段的本质，是在 AI 开始"动手"之前，强制它先"想清楚"，并让人类有机会在低成本的阶段（只是文字计划，没有代码修改）进行纠偏。

### 注意力窗口的物理限制

即使是拥有 200K Token 上下文窗口的模型，在处理大型代码库时也会遇到注意力分散的问题。研究表明，LLM 对上下文开头和结尾的内容注意力最强，中间部分容易被"遗忘"。

在没有规划阶段的大型重构中，AI 在修改第 1 个文件时设定的类型定义，到修改第 5 个文件时可能已经处于上下文的"中间位置"，注意力权重显著下降。

**规划阶段的作用**：把所有关键约束（受影响的文件、接口定义、执行顺序）集中在一个结构化的计划文档里，作为执行阶段的"锚点"，防止 AI 在执行过程中"漂移"。

### 低成本验证门槛

规划阶段的成本通常远低于直接进入执行阶段，但它提供了一个关键的验证机会：

- 如果计划里出现了你不认识的文件路径 → AI 可能在幻觉
- 如果计划里遗漏了关键的依赖文件 → 在执行前就能发现
- 如果计划的执行顺序有问题 → 可以在不修改任何代码的情况下纠正

**类比**：建筑师在施工前画图纸，不是因为图纸比施工更重要，而是因为在图纸上修改的成本远低于在已建好的墙上修改。

## 延伸的习惯

### 计划文件的持久化

对于大型任务，把 AI 生成的计划保存为文件：

```bash
# 让 AI 把计划写入文件
claude "把你的实施计划写入 .claude/plans/oauth2-migration.md"

# 执行阶段开始时，引用这个计划文件
claude "按照 .claude/plans/oauth2-migration.md 的计划执行，
不要超出计划范围"
```

这样做的好处：
- 计划可以被版本控制（`git add .claude/plans/`）
- 团队成员可以审查计划
- 执行阶段可以随时引用计划作为约束

### 子任务分解

对于超大型任务，不要试图在一个计划里覆盖所有内容。把计划分解为多个子任务，每个子任务对应一个独立的会话：

```markdown
# 大任务：将 Express 应用迁移到 Fastify

## 子任务 1（会话 1）：迁移路由层
## 子任务 2（会话 2）：迁移中间件
## 子任务 3（会话 3）：迁移错误处理
## 子任务 4（会话 4）：更新测试
```

每个子任务完成后提交代码，然后开启新会话处理下一个子任务。

### 规划阶段的澄清问题

在规划阶段，鼓励 AI 主动提问。一个好的规划提示词应该包含：

```
"如果有任何不清楚的地方，先向我提问，
不要自行假设。特别是：
- 现有代码的哪些部分是不能修改的？
- 有没有我没有提到但可能受影响的模块？"
```

---

## 个人 / 协作注意

**个人使用**：规划阶段看起来"浪费时间"，但实际上节省了大量的返工时间。一个 5 分钟的规划审查，可以避免 30 分钟的代码回滚和重新开始。

**团队协作**：
- 计划文件可以作为 Code Review 的前置步骤——在 AI 开始写代码之前，团队成员先审查计划
- 对于影响多人工作的重构，计划文件应该纳入版本控制并通知相关人员
- 可以建立团队规范：超过 3 个文件的修改，必须先有 AI 生成的计划文件

---

## 延伸调查方向

1. **规划阶段对输出质量的量化影响**：有没有对比实验，测量"有规划阶段"和"无规划阶段"在大型重构任务中的成功率差异？失败的定义是什么（编译失败？测试失败？逻辑错误？）

2. **不同工具的规划模式实现差异**：Claude Code 的 `plan` permission mode 在底层如何限制修改？Cursor 的 Ask 与 Custom Modes 在规划阶段的边界到底有多稳定？

3. **计划的最优粒度**：计划应该细化到什么程度？"修改第 23-45 行"是否过于细化？"修改认证逻辑"是否过于模糊？有没有关于计划粒度与执行成功率关系的研究？

4. **工具化规划的实际效果**：Codex 的只读 `exec`、Claude Code 的 `plan`、Cursor 的 Ask / Custom Modes，在大型任务上的计划质量和执行偏移率分别如何？
