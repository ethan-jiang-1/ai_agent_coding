# 第二章：三次纠错法则——修改原提示词，而非追加对话

> 本章标注：`I` 为主，工具操作以 `A/B` 校正

## 习惯

**AI 答错了，回退到原提示词编辑后重新运行，而不是在聊天框末尾追加纠正。**

---

## 正确的做法

### 核心操作：Edit & Rerun

当 AI 生成的代码遗漏了某个边界条件，或者方向偏了，正确的操作流程是：

1. **丢弃**生成的错误代码（不要保留在工作区）
2. **向上滚动**到最初的提示词
3. **编辑**原提示词，补充遗漏的约束条件
4. **重新提交**这个被优化的完整提示词

### 各工具的具体操作

**Claude Code（CLI）**
```bash
# 在 CLI 中，用 shell history 找回原始提示词
# 或者直接重新输入一个更完整的版本

# 如果 AI 已经修改了文件，先精确撤销目标文件
git restore --source=HEAD --worktree src/auth/login.py

# 然后重新发送优化后的提示词（一次性包含所有约束）
# 而不是追加"你忘了处理 null 值"
```

**Cursor**
- 在 Composer 中，找到最初的提示词消息
- 点击消息旁边的编辑图标（铅笔）
- 修改提示词内容，点击重新运行
- 如果上一次 agent 已经改了文件，用 checkpoint 或 git 先回到正确边界，再重跑

**Aider（CLI）**
```bash
# Aider 有专门的撤销命令
/undo  # 撤销上一次 AI 的修改

# 然后重新发送更完整的提示词
# 而不是追加纠正
```

**Codex CLI**
```bash
# 一次性工作流：直接重跑一条更完整的命令

# 第一次（方向错了）
codex exec --ephemeral -s workspace-write "帮我写一个用户登录函数"

# 如果 AI 修改了文件但遗漏了 null 处理，先回滚目标文件
git restore --source=HEAD --worktree src/auth/login.ts

# 然后重跑更完整的命令
codex exec --ephemeral -s workspace-write "写一个用户登录函数，必须处理 username/password 为 null 或空字符串的情况，
返回 {error: 'invalid_credentials'}，密码错误不暴露具体原因"

# 如果你在交互式 Codex session 里工作，方向错了也不要不断追加纠正，
# 而是回到原始约束，必要时新开 session，或用 `codex fork --last`
# 从更干净的分支重新执行
```

**注意**：Codex CLI 并不是“只有一次性命令、没有持续对话”。它既支持 `exec --ephemeral` 的重跑，也支持交互式 session、`resume`、`fork`。本章要强调的是：无论你用哪条工作流，方向错了都应回到原始约束重跑，而不是在脏上下文里一条条追加纠正。

### 什么时候追加是可以的

追加纠正并非完全不可用，以下情况是合理的：
- AI 的方向是对的，只是遗漏了一个小细节（追加一句话补充）
- 需要提供 AI 无法预知的运行时信息（如实际的错误日志）
- 对话轮次还很少（前 2-3 轮），上下文还很干净

**判断标准**：如果你发现自己在追加第三条纠正，停下来，重新写原提示词。

## 错误的可能

### 错误一：无休止的追加纠正链

最典型的错误模式：

```
用户：帮我写一个用户登录函数
AI：[生成了代码，但没有处理 null 值]
用户：你忘了处理 null 值
AI：[修改了代码，但引入了新的问题]
用户：不对，这样会破坏现有的测试
AI：[又修改了，但忘了之前的 null 处理]
用户：你又把 null 处理删掉了！
AI：[陷入循环...]
```

**真实后果**：上下文里充满了互相矛盾的指令和中间态代码。AI 在试图同时满足所有追加的约束时，往往会顾此失彼。Reddit 上有大量用户报告这种"AI 在同一个简单错误上陷入死循环"的现象。

### 错误二：保留错误代码继续追加

更危险的情况：AI 生成了错误的代码，用户没有撤销，而是直接追加"这段代码有问题，帮我修"。

**真实后果**：AI 现在需要同时处理：原始需求 + 错误的实现代码 + 用户的纠正指令。这三者之间可能存在矛盾，AI 会尝试"调和"它们，结果往往是生成一个更复杂但仍然错误的版本。

### 错误三：用自然语言描述错误而不是提供精确约束

```
# 错误的追加方式
"这个函数不够健壮，需要更好的错误处理"

# 正确的原提示词修改方式
"写一个用户登录函数，需要：
1. 处理 username 或 password 为 null/空字符串的情况，返回 {error: 'invalid_credentials'}
2. 处理数据库连接失败的情况，抛出 DatabaseError
3. 密码验证失败时不暴露具体原因（防止用户枚举攻击）"
```

模糊的纠正（"更健壮"、"更好的错误处理"）给了 AI 太多解释空间，往往导致过度工程化。

## 背后的原理

### 负向 Few-shot 学习

LLM 的工作方式是从上下文中识别模式，然后生成符合这些模式的输出。当上下文里充满了错误的代码和失败的推理过程时，模型实际上在学习"在这个项目里，这类问题的答案看起来是这样的"——即使那些答案是错的。

**类比**：想象你在教一个学生解数学题。如果你把所有的错误尝试都留在草稿纸上，学生在解下一道题时会被这些错误的思路干扰。最好的做法是擦掉错误的尝试，从一张干净的纸开始。

### 对话树的扁平化

从 AI 的视角来看，一个经过多次追加纠正的对话，和一个包含完整约束的单次提示词，处理难度完全不同：

- **多次追加**：AI 需要在脑中"合并"所有的指令，解决它们之间的矛盾，同时还要考虑已经生成的中间代码
- **单次完整提示词**：AI 只需要处理一个清晰的需求，没有历史包袱

**数学上**：追加纠正每增加一轮，AI 需要处理的约束关系数量呈指数增长（N 个约束之间有 N² 个潜在矛盾需要解决）。

### Token 的信息密度

追加纠正的对话历史，Token 消耗高但信息密度低。一个包含 5 轮追加纠正的对话，可能消耗了 10,000 个 Token，但其中真正有用的信息（最终的需求约束）可能只需要 500 个 Token 就能表达清楚。

## 延伸的习惯

### 提示词版本管理

对于复杂任务，把提示词当作代码来管理：

```markdown
# 在项目根目录维护 prompts/ 目录
prompts/
├── auth_login_v1.md    # 第一版，太模糊
├── auth_login_v2.md    # 第二版，加了边界条件
└── auth_login_v3.md    # 第三版，加了安全约束（当前使用）
```

这样做的好处：
- 下次遇到类似任务，可以直接复用优化过的提示词
- 团队成员可以共享和改进提示词
- 提示词本身成为项目文档的一部分

### 约束条件的结构化

在修改原提示词时，使用结构化的格式而不是自然语言段落：

```markdown
# 好的提示词结构
任务：写一个用户登录函数

必须满足：
- [ ] 处理 null/空字符串输入
- [ ] 密码验证失败不暴露具体原因
- [ ] 数据库连接失败时抛出 DatabaseError

禁止：
- 使用 console.log 调试输出
- 修改现有的 User 模型结构

参考：@src/auth/register.ts（保持风格一致）
```

这种格式让 AI 可以逐条检查，减少遗漏。

### 错误代码的及时清理

在重新提交提示词之前，确保工作区是干净的：

```bash
# 检查 AI 修改了哪些文件
git diff --name-only

# 精确撤销特定文件
git restore --source=HEAD --worktree src/auth/login.py

# 如果确实要批量撤销，先确认 diff，再按目录或文件粒度恢复
# 不要下意识用整个仓库范围的粗暴回滚
```

## 个人 / 协作注意

**个人使用**：很多人有"沉没成本"心理——觉得之前的对话历史是有价值的，不舍得放弃。但实际上，一个干净的重新开始几乎总是比在混乱的上下文里继续挣扎更快。

**团队协作**：如果团队共享提示词库（prompts/ 目录），需要约定提示词的版本管理规范。特别是对于高频使用的任务（如代码审查、测试生成），一个经过优化的提示词模板可以让整个团队受益。

---

## 延伸调查方向

1. **追加纠正 vs 重新提示的输出质量对比**：有没有系统性的实验数据，比较同一任务在"追加 3 次纠正"和"重写原提示词"两种方式下的输出质量？

2. **不同工具的 Edit & Rerun 实现差异**：Cursor 的编辑重跑与 checkpoint 如何配合？Claude Code 与 Codex CLI 在“重跑原始约束”时的最省成本路径分别是什么？这些实现细节值得深入测试。

3. **提示词版本管理的最佳实践**：有没有专门的工具或框架来管理 AI 提示词的版本？与代码版本控制的集成方式？

4. **"三次"的科学依据**：三次纠错法则是经验值。有没有研究表明，在第几次追加纠正后，输出质量开始显著下降？这个阈值是否因任务类型而异？
