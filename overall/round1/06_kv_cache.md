# 第六章：KV Cache 经济学——让稳定上下文摊薄成本

## 习惯

**把不变的内容（规则文件、系统 Prompt、Repo Map）放在对话的最前面，利用提示词缓存机制，让"固定成本"被后续每轮对话以极低边际成本复用。**

---

## 正确的做法

### 理解缓存命中的条件

KV Cache 的核心逻辑：如果你发送给模型的内容前缀与上一次完全一致，模型就不需要重新计算这部分，直接复用上次的计算结果。

**命中缓存的必要条件**：
1. 内容完全相同（一个字符的差异都会导致缓存失效）
2. 内容位置在对话的开头（前缀匹配）
3. 内容超过最小缓存阈值（Anthropic: ~1024 Token；Gemini: 1024 Token）

**实际操作原则**：
- 稳定内容（规则、约束、架构说明）放在每次对话的最前面
- 变动内容（当前任务、具体问题）放在最后面
- 不要在稳定内容里夹杂任何会变化的信息

### 各工具的缓存利用方式

**Aider（CLI）——最显式的缓存利用**
```bash
# --read 参数加载的文件会被标记为只读并优先缓存
aider --read CONVENTIONS.md --read docs/architecture.md

# Aider 在使用 Anthropic API 时，会自动对 CONVENTIONS.md
# 和 Repo Map 进行缓存标记（cache_control: ephemeral）
# 这些内容只在第一轮对话时全量计费，后续命中缓存价格降至 10%
```

**Claude Code（CLI）**
```bash
# CLAUDE.md 在每次会话启动时自动加载，且位置固定
# 只要 CLAUDE.md 内容不变，就能持续命中缓存

# 错误示范：频繁修改 CLAUDE.md（每次修改都会导致缓存失效）
# 正确示范：保持 CLAUDE.md 稳定，把任务相关信息放在对话里
```

**Gemini CLI**
```bash
# Gemini 2.5 默认对超过 1024 Token 的输入启用隐式缓存
# 开发者不需要显式配置，只需保持上下文前缀稳定
gemini "修改 src/auth/login.ts 的 null 处理逻辑"
# ↑ 系统提示词（GEMINI.md）+ 项目索引 会自动缓存
```

**Cursor / Windsurf（IDE）**
- 这些工具通常在内部处理缓存，用户不直接控制
- 但规则：保持 .cursorrules 内容稳定，不要频繁修改
- Windsurf 的 Fast Context（RAG 系统）会缓存代码库的语义索引

**Codex CLI**
```bash
# Codex CLI 底层调用 OpenAI API，同样支持提示词缓存
# codex.md（全局规则文件）在每次命令启动时自动加载
# 保持 codex.md 内容稳定 = 持续命中缓存前缀

# 注意：Codex CLI 每次命令是独立会话
# 缓存收益主要来自 codex.md 的多次重复加载，而非会话内的历史累积
# 规则文件越精简、越稳定，每次命令启动的缓存命中率越高
```

### 结构化对话模板：稳定在前，变动在后

```
[稳定部分 → 缓存命中]
---
项目规则：[引用 CLAUDE.md 内容]
架构约束：[引用 architecture.md]
技术栈：TypeScript, Node.js 20, Vitest
---

[变动部分 → 每次重新计算，但 Token 数量少]
当前任务：修改 src/auth/login.ts 中的 null 处理逻辑
具体要求：当 username 为空字符串时，返回 400 而不是 500
```

## 错误的可能

### 错误一：在稳定内容里混入动态信息

```markdown
# 错误的 CLAUDE.md（包含动态内容，每次都不同）
## 项目规则
- 使用 TypeScript
- 当前日期：2026-04-07（❌ 这会导致每天缓存失效）
- 最近修改的文件：src/auth/login.ts（❌ 每次任务都不同）

# 正确的 CLAUDE.md（纯静态内容）
## 项目规则
- 使用 TypeScript
- 包管理器：pnpm
- 测试框架：Vitest
```

一个字符的变化就会让整个前缀的缓存失效。把任何可能变化的信息（日期、当前任务、文件状态）从稳定层移出。

### 错误二：把稳定内容放在对话末尾

```
# 错误的顺序
[当前任务描述]  ← 变动内容在前
[项目规则]      ← 稳定内容在后（无法命中缓存）

# 正确的顺序
[项目规则]      ← 稳定内容在前（可以命中缓存）
[当前任务描述]  ← 变动内容在后
```

KV Cache 是**前缀匹配**——只有从头开始完全相同的部分才能命中缓存。把稳定内容放在后面，等于放弃了缓存收益。

### 错误三：默认使用 1M Token 模型处理小项目

Anthropic 近期悄然将 Claude Code 的默认模型切换为支持 1M Token 的扩展变体。许多开发者未察觉这一变化，在小项目中也使用了 1M Token 的上下文窗口，导致每次微小的代码修改都附带了庞大的历史负载。

**真实后果**：用户报告在 5 小时内迅速烧穿使用额度，但任务量并没有增加。原因是 1M 上下文版本在每次请求时发送了远超必要的上下文负载。

**解决方案**：
```bash
# 在 ~/.claude/settings.json 中降级回标准版本
{
  "model": "claude-sonnet-4-6",  # 而不是 1M 版本
  "maxTokens": 200000            # 明确限制上下文窗口
}
```

### 错误四：频繁修改规则文件导致缓存持续失效

有些团队把 CLAUDE.md 当作"今日任务说明"来用，每天早上修改里面的内容。这让这个本来应该是"稳定前缀"的文件变成了"每天都不同的变量"，完全失去了缓存收益。

## 背后的原理

### KV Cache 的工作机制

Transformer 模型处理 Token 的过程，本质上是为每个 Token 计算 Key（K）和 Value（V）向量，然后通过注意力机制计算它们之间的关系。这个计算过程非常耗时，也是 API 调用延迟的主要来源。

**缓存的本质**：当输入的前缀完全一致时，这部分 K、V 向量不需要重新计算，直接从缓存中读取。

**类比**：想象你每天上班都需要走同一条路。如果你每天都记录一遍路线图，那太浪费时间了。更好的做法是把路线图贴在门口（固定位置），每次出发前直接查看，不需要重新绘制。

KV Cache 就是这个"贴在门口的路线图"——固定位置、固定内容，每次使用不需要重新生成。

### 成本结构对比

以 Anthropic Claude 为例：

| 计费类型 | 价格（每百万 Token） | 适用情况 |
|---------|---------------------|---------|
| 标准 Input Token | $3.00 | 未命中缓存的内容 |
| 缓存写入（Cache Write） | $3.75 | 第一次建立缓存时 |
| 缓存读取（Cache Read） | $0.30 | 命中缓存后的每次读取 |
| Output Token | $15.00 | AI 生成的内容 |

**经济学计算**：
- 一个 2000 Token 的 CLAUDE.md，每次会话使用
- 不使用缓存：$0.006/次
- 第一次建立缓存：$0.0075（略贵）
- 后续每次命中缓存：$0.0006（便宜 10 倍）
- **只需 2 次命中缓存，就能回收第一次的额外成本**

对于一个团队每天使用 100 次的规则文件，缓存的年节省额非常可观。

### 文档越详尽，长期越省钱

这是一个反直觉的结论：在使用了 KV Cache 的情况下，规则文件越详尽（在合理范围内），长期来看越省钱——因为固定成本（缓存写入）只发生一次，而边际成本（缓存读取）接近于零。

**前提**：详尽的规则文件内容必须是真正有价值的、稳定的约束，而不是为了"看起来全面"的填充内容。

## 延伸的习惯

### 显式缓存标记（Claude API 直接调用时）

如果你通过 API 直接调用 Claude（例如在自建工具中），可以使用显式缓存标记：

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "你是一个 TypeScript 专家...",
            "cache_control": {"type": "ephemeral"}  # 标记这部分为可缓存
        }
    ],
    messages=[
        {"role": "user", "content": "帮我修改这个函数..."}
    ]
)
```

### 会话内的缓存节奏

在一个长会话中，每一轮对话都会在上下文末尾追加新内容。随着会话变长，可缓存的前缀比例下降，缓存收益减少。这是"长会话要及时压缩"的另一个经济学理由：

```
第 1 轮：[CLAUDE.md 2000 Token][任务 200 Token] → 缓存率 90%
第 10 轮：[CLAUDE.md 2000 Token][历史 8000 Token][任务 200 Token] → 缓存率 20%
第 30 轮：[CLAUDE.md 2000 Token][历史 25000 Token][任务 200 Token] → 缓存率 7%
```

使用 /compact 压缩历史，既节省 Token 成本，又恢复了缓存效率。

### Aider 的缓存最佳实践

```bash
# 把所有只读参考文件通过 --read 加载（而不是 /add）
# --read 的文件会被优先缓存，且 AI 不会尝试修改它们
aider \
  --read CONVENTIONS.md \        # 代码规范（稳定，缓存）
  --read docs/architecture.md \  # 架构文档（稳定，缓存）
  src/auth/login.ts              # 要修改的文件（变动）
```

## 个人 / 协作注意

**个人使用**：建立"稳定内容不轻易改动"的纪律。每次修改 CLAUDE.md 或 .cursorrules 都有成本——不是修改本身，而是缓存失效导致的成本恢复期。

**团队协作**：
- 团队共享的规则文件（CLAUDE.md）的修改，应该像修改基础设施配置一样谨慎，需要评审
- 个人偏好的修改（CLAUDE.local.md）不影响团队缓存
- 可以计算团队的缓存节省额，作为维护规则文件稳定性的激励：每天 N 次调用 × M Token 规则文件 × 90% 缓存率 × $2.70/百万 Token 节省额

## 延伸调查方向

1. **实测缓存命中率**：在真实的 AI Coding 工作流中，CLAUDE.md 的缓存命中率大约是多少？随着会话变长，命中率如何下降？有没有用户做过系统性测试？

2. **不同工具的缓存透明度**：Cursor、Windsurf 等 IDE 工具是否向用户暴露缓存命中信息？有没有办法观察到缓存在实际工作中的节省效果？

3. **Gemini 隐式缓存的实际行为**：Gemini 的隐式缓存（Implicit Caching）在什么条件下会触发？1024 Token 的阈值是否可以配置？缓存的生命周期是多长？

4. **1M Token 模型的隐性成本**：Claude Code 切换到 1M Token 默认变体后，在标准开发任务中，实际发送的上下文大小是多少？与 200K 版本相比，成本差异有多大？
