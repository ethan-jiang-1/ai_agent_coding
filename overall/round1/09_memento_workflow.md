# 第九章：跨会话状态传递——Memento 工作流

## 习惯

**会话上下文达到 70% 时，主动生成状态快照文件；新会话开始时，先读取快照恢复状态，而不是重新扫描整个代码库。**

---

## 正确的做法

### 为什么需要 Memento 工作流

LLM 是无状态的——每次新会话，它对之前发生的一切一无所知。这意味着在长期项目中，你需要主动管理"AI 的记忆"。

Memento 工作流的名字来自同名电影：主角因为丧失短期记忆，必须依靠不断记录笔记来向未来的自己传递信息。AI Coding 的长期项目管理，本质上是同样的逻辑。

### 四个核心组件

**1. 会话交接文档（session-handoff.md）**

在会话结束前生成：
```bash
# 在 Claude Code 中
claude "生成一份会话交接文档，包含：
1. 本次会话完成了什么
2. 做了哪些重要的架构决策，以及原因
3. 遇到了哪些问题，如何解决的
4. 下一个会话需要从哪里继续
5. 需要注意的坑和约束

把结果写入 .claude/handoffs/2026-04-07-auth-module.md"
```

**2. /compact 命令（Claude Code 内置）**

```bash
# 在上下文使用量约 70% 时主动触发
/compact

# 效果：Claude Code 在后台将冗长的会话历史压缩为语义摘要，
# 保留关键决策和当前状态，清除冗余的试错记录
# 可以在同一会话内继续工作，上下文开销大幅减少
```

**3. Memory Bank 架构（Cline / Roo Code）**

Cline 的显式记忆系统，维护一组结构化文件：

```
memory-bank/
├── activeContext.md    # 当前工作状态和即时优先事项
├── progress.md         # 里程碑、已完成的工作、待办
├── productContext.md   # 产品需求和用户故事
├── systemPatterns.md   # 架构决策和代码模式
└── techContext.md      # 技术栈、依赖、环境配置
```

**使用方式**：
```
# 会话结束前
"更新 memory-bank/ 目录，记录本次会话的工作状态"

# 新会话开始时
"首先读取 memory-bank/ 目录下的所有文件，恢复项目上下文，然后..."
```

**4. Claude Code Auto-Memory（自动记忆）**

Claude Code 的自动化版本，不需要手动管理：
```
# ~/.claude/projects/<project>/memory/ 目录
# Claude Code 自动将以下内容记录到这里：
# - 构建命令和依赖关系
# - 调试过程中发现的特殊情况
# - 特定的代码风格偏好（经过纠正后）
# - 项目特定的约束条件

# 每次新会话时，自动读取前 200 行或 25KB 内容注入上下文
```

### 新会话的正确启动方式

```bash
# 错误的启动方式（让 AI 重新探索整个代码库）
claude "我们继续昨天的 OAuth2 迁移工作"
# AI 不知道昨天做了什么，会重新扫描整个项目

# 正确的启动方式（先恢复状态，再继续工作）
claude "读取 .claude/handoffs/2026-04-07-auth-module.md，
了解上次会话的进度，然后继续 OAuth2 迁移工作，
从中间件层开始（上次完成了配置层）"
```

**关键差异**：前者让 AI 用几千 Token 重新"发现"项目状态，后者直接给 AI 一个高密度的状态摘要，新会话第一轮就能有效工作。

## 错误的可能

### 错误一：靠 AI"自己记住"跨会话的决策

```
# 错误的期待
用户：（新会话）我们继续昨天讨论的架构方案
AI：我没有关于昨天对话的记忆。请告诉我你指的是哪个架构方案？
用户：就是昨天那个！用 Event Sourcing 的那个！
AI：（只能基于有限信息猜测，很可能方向跑偏）
```

这不是 AI 的问题，而是用户对 AI 记忆机制的误解。每个新会话，AI 的上下文是完全空白的。

**真实代价**：在一个复杂项目中，每次新会话都需要花 10-20 分钟重新"带 AI 入门"。如果每天开 3-4 个会话，这个开销每天可能超过 1 小时，还不包括因为 AI 状态恢复不完整导致的错误决策。

### 错误二：在上下文 95% 时才被动触发压缩

```
# 错误的时机：等到系统强制压缩
上下文使用量：95%
系统提示：上下文即将达到限制，正在自动压缩...
# 此时 AI 的推理能力已经因为上下文过满而显著下降
# 自动压缩的质量通常低于主动压缩

# 正确的时机：70% 时主动触发
上下文使用量：70%
用户：/compact 或 "生成会话交接文档"
# AI 的推理能力仍然正常，能生成高质量的状态摘要
```

### 错误三：快照文件没有纳入版本控制

```bash
# 常见的 .gitignore 错误
.claude/handoffs/  # 不应该被忽略！
memory-bank/       # 这些文件是项目状态的一部分

# 正确的 .gitignore 配置
.claude/settings.local.json    # 个人设置，不共享
CLAUDE.local.md                # 个人偏好，不共享
# 但 memory-bank/ 和 .claude/handoffs/ 应该纳入版本控制
```

团队成员的会话交接文档如果不纳入版本控制，就失去了跨成员共享上下文的价值。

### 错误四：快照文件过于详细，反而失去价值

```markdown
# 糟糕的快照（太长，信息密度低）
## 本次会话工作记录
我首先打开了 src/auth/login.ts，然后发现第 23 行有一个问题...
接着我尝试了第一种方案，使用 JWT 验证，结果发现...
然后我尝试了第二种方案...
（500 行的详细记录）

# 好的快照（简洁，高密度）
## 状态：OAuth2 迁移（进行中）
### 已完成
- 配置层（src/config/auth.config.ts）✓
### 当前进度
- 中间件层（src/auth/middleware.ts）进行中，约 60% 完成
### 关键决策
- 选择了 PKCE 流程而不是隐式授权（原因：安全合规要求）
### 下一步
- 完成中间件层的 token 刷新逻辑（第 45-67 行）
### 注意事项
- 不要修改 User 模型（产品要求，已确认）
```

一个好的快照应该能在 30 秒内让任何人（包括 AI）理解当前状态。

## 背后的原理

### LLM 的无状态本质

大语言模型在数学上是一个函数：`f(context) → response`。函数本身没有记忆，只有输入（context）决定输出。

这意味着：
- 不同会话之间，模型完全不知道之前发生了什么
- 同一会话中，模型"记得"之前的内容，是因为那些内容在当前的上下文里
- 上下文就是 AI 的全部"工作记忆"

Memento 工作流的本质：把本应在 AI 脑子里的"工作记忆"，外化到文件系统里，使其可以持久化、可以跨会话传递。

### 压缩的信息密度

50,000 Token 的试错对话，真正有价值的信息可能只需要 500 Token 表达：
- 最终决定采用方案 A（而不是 B 和 C）
- 方案 A 的具体实现要点
- 遇到的一个关键限制条件

把 50,000 Token 的历史压缩成 500 Token 的摘要，不只是节省了 Token，更重要的是提升了信息密度。新会话的 AI 读取 500 Token 的高密度摘要，往往比读取 50,000 Token 的完整历史"理解"得更深——因为噪音被去除了。

**类比**：一本 500 页的设计评审记录，和一份 5 页的设计决策摘要。后者对于新加入的工程师来说，理解项目历史的效率往往更高。

### 70% 的最优压缩时机

为什么是 70% 而不是 90% 或 50%？

- **过早（50%）**：会话刚进入状态，频繁压缩打断工作流
- **过晚（90%+）**：AI 的推理能力因上下文过满已经明显下降，压缩质量不佳
- **70%**：AI 仍然在最佳状态，能生成高质量的压缩摘要，且还有足够空间继续工作

## 延伸的习惯

### 建立标准化的快照模板

```markdown
<!-- .claude/templates/handoff-template.md -->
# 会话交接：[日期] [功能名称]

## 当前状态
- [ ] 任务完成度：___%
- 所在阶段：探索 / 规划 / 实现 / 测试 / 完成

## 已完成
- （列出已完成的具体工作）

## 进行中
- （当前正在做什么，停在哪里）

## 关键决策
| 决策 | 原因 | 替代方案 |
|------|------|---------|

## 下一步（优先级排序）
1. 
2. 

## 不要碰的地方
- （明确列出不应该修改的文件或模块）

## 环境信息
- 分支：
- 上次测试结果：
```

### 利用 Git Commit 作为状态快照

每次完成一个逻辑单元，都做一个带有详细信息的 Git commit：

```bash
git commit -m "feat(auth): 完成 OAuth2 配置层

- 添加了 PKCE 流程配置
- 选择不使用隐式授权（安全合规要求）
- 下一步：middleware.ts 的 token 刷新逻辑

AI 辅助完成，会话交接文档：.claude/handoffs/2026-04-07.md"
```

Git 历史本身就是一个结构化的状态快照序列。

## 个人 / 协作注意

**个人使用**：建立"会话结束前 5 分钟生成快照"的固定习惯。不需要完美，能在下次会话开始时节省 10 分钟就是值得的。

**团队协作**：
- 快照文件纳入版本控制，成为异步协作的基础设施
- 当 A 把一个任务交接给 B 时，AI 的会话快照比口头说明更准确、更完整
- 团队共享的 memory-bank/ 相当于一个"项目 AI 记忆"——任何人在任何时候启动新会话，都能快速获取项目状态
- 明确约定快照文件的命名规范（日期 + 功能名 + 作者），避免混乱

## 延伸调查方向

1. **快照质量 vs 人工重新描述的对比**：用 AI 生成的会话快照 vs 人工写的任务说明来启动新会话，哪种方式在第一轮的输出质量上更好？信息密度有多大差异？

2. **Claude Code Auto-Memory 的实际行为**：Auto-Memory 机制具体记录了哪些类型的信息？记录的质量如何？与手动生成的快照相比，哪种方式更有效？

3. **Cline Memory Bank 的团队采用情况**：在多人团队中使用 Memory Bank 模式，实际效果如何？有没有关于团队内 AI 上下文共享效率提升的案例？

4. **/compact 的压缩质量**：Claude Code 的 /compact 命令会丢失哪些信息？压缩后的摘要能覆盖原始会话多少比例的有效信息？有没有对比主动 /compact 和被动等待的质量差异？
