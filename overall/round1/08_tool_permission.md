# 第八章：工具权限克制——联网与 MCP 按需开关

## 习惯

**默认关闭联网搜索和非必要 MCP 工具。只在明确诊断出本地知识盲区时，短暂开启特定工具，获取信息后立即关闭。**

---

## 正确的做法

### 权限最小化原则

给 AI 的工具权限越多，它的行为越难以预测和控制。正确的工具权限管理遵循以下层次：

```
默认状态（日常使用）：
  ✓ 本地文件读取
  ✓ 本地文件修改（限定范围）
  ✓ 终端命令（受限）
  ✗ 联网搜索（关闭）
  ✗ MCP 服务器（关闭）

按需开启（特定任务）：
  → 调试生产问题：开启 Sentry/Honeycomb MCP
  → 集成新 API：开启对应的 OpenAPI MCP
  → 需要最新文档：开启 Web Search（时间限定）
  ✓ 任务完成后立即关闭
```

### 各工具的权限控制方式

**Claude Code（CLI）**
```bash
# 启动时禁止网络访问
claude --no-web-search "修改 src/auth/login.ts"

# 临时开启特定 MCP（调试阶段）
claude --mcp sentry "分析生产环境的 500 错误"
# 调试完成后，重新启动不带 MCP 参数的会话

# 限制文件修改范围（防止 AI 超出范围）
claude "只修改 src/auth/ 目录下的文件，不要碰其他目录"
```

**Aider（CLI）**
```bash
# Aider 的 /add 命令精确限定可修改的文件
# 只 /add 真正需要修改的文件，物理上隔绝了 AI 修改其他文件的可能

/add src/auth/login.ts        # 只允许修改这一个文件
# 不要 /add .（整个目录）

# 禁止 Aider 自动执行 Shell 命令
aider --no-auto-commits       # 生成代码但不自动提交
# 手动审查后再 git add && git commit
```

**OpenCode**
```json
// opencode.json：工具白名单配置
{
  "agents": {
    "default": {
      "tools": ["read_file", "write_file", "run_command"],
      // 注意：没有 "web_search" 和 "mcp_*"
      "maxToolCalls": 20  // 防止死循环
    }
  }
}
```

**Cursor**
```
# 在 .cursorrules 中明确禁止
"不要使用 Web Search 来查找 API 文档，
 只参考 @docs/ 目录下的本地文档"

# 或者在提示词中每次明确
"只使用本地代码库的信息，不要搜索外部资源"
```

### MCP 的正确使用模式

**按场景连接，场景结束后断开**：

```
调试场景：
  1. 出现生产问题 → 连接 Sentry MCP
  2. AI 查询错误链路 → 找到根本原因
  3. 修复代码 → 断开 Sentry MCP
  4. 继续正常开发（无 MCP）

API 集成场景：
  1. 需要集成 Stripe → 连接 Stripe OpenAPI MCP
  2. AI 查询端点规范 → 生成准确的接口调用代码
  3. 集成完成 → 断开 Stripe MCP
  4. 继续正常开发（无 MCP）
```

**不要做的事**：在整个开发周期内保持所有 MCP 服务器常开。如果 AI 在处理无关任务时"顺手"调用了高权限的 MCP 工具（如数据库写入 MCP、CI/CD 触发 MCP），后果可能是严重的。

## 错误的可能

### 错误一：默认开启 Web Search，让 AI 自行搜索文档

**真实后果**：

场景：用户让 AI 集成 Stripe 支付接口，AI 通过 Web Search 搜索文档。

搜索结果可能包含：
- 2021 年的博客教程（基于 Stripe API v1，已经弃用）
- Stack Overflow 回答（基于用户误解的用法）
- 非官方的第三方包装库（不是官方 SDK）

AI 会自信地基于这些过时信息生成代码。代码语法正确，但在运行时报 404——因为那个 API endpoint 已经不存在了。

这种错误很难调试，因为 AI 生成的代码"看起来很对"。

### 错误二：工具调用死循环（ReAct 架构的陷阱）

基于 ReAct（推理-行动循环）架构的 AI，当无法获取所需信息时，会反复尝试工具调用：

```
AI 的内部循环：
1. 尝试 read_file("src/config.ts") → 文件不存在
2. 尝试 search_files("config") → 找到 10 个文件
3. 逐一读取 → 没找到需要的内容
4. 尝试 web_search("project config pattern") → 获取通用信息
5. 再次尝试 read_file → 循环...
```

**历史案例**：AutoGPT 在 2023 年曾展示过这种风险——系统陷入无限的浏览器打开和 API 调用循环，烧掉大量资金却没有完成任何目标。这个案例后来成为 AI 工具设计中"必须有工具调用上限"的标准论据。

**防止措施**：
- 设置每次会话的最大工具调用次数（如 OpenCode 的 maxToolCalls: 20）
- 当 AI 连续 3 次工具调用没有进展时，手动介入
- 不要给 AI 开放过多工具，让它"自由探索"

### 错误三：MCP 权限蔓延（在无关任务中意外使用高权限工具）

```
场景：
- 调试阶段连接了数据库写入 MCP（为了修复数据问题）
- 调试完成后，没有断开 MCP
- 后续让 AI 做无关的代码格式化任务
- AI 在"优化"过程中，意外通过 MCP 修改了生产数据库的某条记录
```

这听起来像极端案例，但在 MCP 权限管理不严格的环境里，类似的意外实际上时有发生。

**核心原则**：高权限工具（数据库写入、CI/CD 触发、生产环境访问）绝对不能"常开"。

### 错误四：允许 AI 在不受限的代码库中"自由探索"

没有通过 /add（Aider）或路径限制来界定工作范围：

```bash
# 危险的做法
aider "优化一下数据库查询性能"
# AI 可能读取并修改：src/db/, src/models/, src/api/, src/config/...
# 你无法预知 AI 会碰哪些文件

# 安全的做法
aider src/db/queries/user-queries.ts
# AI 只能修改这一个文件
# 如果需要参考其他文件，用 --read（只读）
```

## 背后的原理

### ReAct 架构的隐性成本

AI Coding 工具通常基于 ReAct（Reason + Act）架构：AI 推理当前状态 → 选择工具 → 执行工具 → 观察结果 → 再次推理 → 再次选择工具...

**关键细节**：每次工具调用都触发一次完整的模型推理过程（Inference Pass）。如果 AI 连续调用 10 次工具，就触发了 10 次推理，成本是单次调用的 10 倍以上（因为每次推理的上下文都在增长）。

**类比**：想象你雇了一个助手，每次他需要查资料，都要先开会讨论"是否需要查"，然后去图书馆，回来后再开会讨论"查到了什么"，然后再决定下一步。工具调用越多，会议越多，效率越低。

给 AI 更少的工具，迫使它更多地依赖本地已有的信息，反而能减少不必要的工具调用循环。

### 网页内容的信息密度问题

一个典型的 API 文档网页，其 HTML 源码中：
- 实际 API 内容：约 20-30%
- 导航栏、页脚、广告：约 30-40%
- JavaScript、CSS 代码：约 30-50%

当 AI 通过 Web Search 抓取并解析网页时，这些噪音会被转化为 Token 塞入上下文，稀释了有效信息的比例。

相比之下，llms.txt 格式的文档，有效信息密度接近 100%。

### 联网工具的时效性悖论

联网能力的本意是弥补 AI 训练数据的时效性缺陷。但实际上：
- AI 搜索到的网页，也可能是 1-2 年前写的
- SEO 排名高的内容不代表信息准确和及时
- 对于快速迭代的框架，搜索结果可能比训练数据更旧

**结论**：联网不是时效性问题的银弹，而是引入了新的不确定性。精确的 MCP 接入（直接连接官方 API 规范）才是解决时效性问题的正确方式。

## 延伸的习惯

### 为工具调用设置上限

```json
// Claude Code 的工具调用限制（通过 CLAUDE.md 或 settings.json）
{
  "maxToolCalls": 30,  // 每次会话最多 30 次工具调用
  "toolCallTimeout": 60  // 单次工具调用超时 60 秒
}
```

当 AI 接近工具调用上限时，强制要求它总结当前状态并向人类汇报，而不是继续盲目尝试。

### 沙箱环境用于危险操作

对于需要执行危险命令（如数据库迁移、文件批量删除）的任务，使用沙箱环境：

```bash
# Codex CLI 提供 OS 级别的沙箱隔离
# macOS: Seatbelt / Linux: Landlock
codex --sandbox "运行数据库迁移测试"
# AI 在隔离环境中执行，无法影响主系统

# 或者使用 Docker 容器作为沙箱
docker run --rm -v $(pwd):/workspace ai-sandbox
```

### 定期审查 AI 工具的权限配置

```bash
# 每月检查一次：哪些 MCP 服务器是常开的？
# 是否还需要它们？
cat .claude/mcp.json
# 或者
cursor settings → MCP
```

## 个人 / 协作注意

**个人使用**：建立"关闭联网是默认状态"的思维定势。每次开启联网工具，都是一个需要思考的决策，而不是理所当然的默认行为。

**团队协作**：
- 在团队 AI 配置中禁用默认联网，需要使用时手动开启并记录原因
- 高权限 MCP（数据库写入、生产环境访问）只能在特定受控场景下开启，需要审批
- 建立"AI 工具安全规范"，明确哪些操作需要人工确认，哪些可以 AI 自主执行

## 延伸调查方向

1. **工具调用死循环的发生频率**：在真实的 AI Coding 工作流中，工具调用死循环发生的频率有多高？不同工具（Claude Code vs Cursor vs Aider）在防止死循环方面的机制有什么差异？

2. **Web Search vs MCP OpenAPI 的集成质量对比**：在 API 集成任务中，系统性地比较"AI 通过 Web Search 获取文档"和"通过 MCP 接入 OpenAPI 规范"两种方式的代码准确率。哪些 API 有高质量的 MCP 支持？

3. **MCP 权限蔓延的实际风险**：有没有关于 MCP 权限管理不当导致意外操作的真实案例？企业级 AI Coding 部署中如何处理高权限工具的访问控制？

4. **Aider /add 精确限定 vs 开放目录的对比**：在真实项目中，精确 /add 单个文件 vs /add 整个目录，对 AI 的修改范围和代码准确率有多大影响？超出预期的修改频率有多高？
