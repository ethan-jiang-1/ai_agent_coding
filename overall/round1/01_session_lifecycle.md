# 第一章：会话生命周期——任务边界即会话边界

## 习惯

**一个逻辑任务完成后，立即开新会话。不要把多个无关任务堆在同一个窗口里。**

---

## 正确的做法

### 定义"任务边界"

任务边界不是时间边界，而是逻辑边界：

- 一个功能模块开发完成并提交 → 新会话
- 一个 Bug 修复并验证通过 → 新会话
- 一次大规模重构完成 → 新会话（即使还有后续工作）
- 连续三次纠错失败 → 立刻重置，不要继续追加

### 各工具的具体操作

**Claude Code（CLI）**
```bash
# 一个逻辑任务完成后，直接结束当前 session
exit

# 新任务开启新会话
claude

# 只有在明确要延续同一任务时，才使用恢复能力
claude --continue
claude --resume
claude --resume <session-id> --fork-session
```

**Cursor**
- 完成一个逻辑任务后，新开一个新的 chat / composer 线程
- 不要在同一个线程里从"修 CSS"切换到"重构数据库层"
- Ask / Agent 都要遵守同样的任务边界

**Aider（CLI）**
```bash
# 清空对话历史，但保留当前文件范围
/clear

# 或者直接退出
/exit

# 新任务重启时，只加入下一个任务真正需要的文件
aider src/auth/login.py
# 不要直接 aider . 让整个目录进入编辑范围
```

**Codex CLI**
```bash
# 一次性任务：用非交互式、无持久化 session
codex exec --ephemeral -s workspace-write "实现 src/auth/login.ts 的 OAuth2 登录逻辑"

# 需要持续上下文时，用交互式 session
codex

# 只有在同一逻辑任务上继续工作时，才恢复或分叉旧会话
codex resume --last
codex fork --last
```

**关键点**：Codex CLI 不是“天然只有一次性新会话”。它同时支持一次性 `exec`，也支持持续 session 的 `resume` / `fork`。正确做法是按任务边界选工作流，而不是默认把所有后续工作都接在同一个旧 session 上。

### 基于时间的定期重置

即使任务没有明确完成，以下情况也应该主动重置：
- 会话持续超过 60-90 分钟
- 工具已经给出明显的上下文压缩 / 历史过长 / 记忆接近上限提示
- 进行了大规模文件重构（AI 缓存的文件结构已经失效）

---

## 错误的可能

### 错误一：大杂烩会话（Kitchen Sink Session）

最常见的错误。一个会话里，用户先让 AI 修改前端 CSS 样式，然后要求重构后端数据库迁移脚本，接着又开始配置 CI/CD 流水线。

**真实后果**：在一个包含 50 个文件的项目中，一个简单的问题可能会拉取高达 18,000 个 Token 的无关代码。这就像要求水管工修理水槽，而对方却开始阅读整栋房屋的建筑蓝图。

**为什么会发生**：AI 在处理新问题时，会尝试从上下文中寻找相关信息。上下文里有什么，它就会"联想"到什么。前面的 CSS 修改历史会干扰后面的数据库重构判断。

### 错误二：在已经混乱的会话里继续纠错

当 AI 开始给出奇怪的答案时，用户的本能反应是继续追加纠正："不对，你忘了处理 null 值"、"还是不对，再试一次"……

**真实后果**：上下文里充满了被否定的旧方案、冗长的错误日志和反复尝试的中间代码。AI 会从这些失败的历史中"学习"，陷入越来越深的错误循环。Reddit 上有大量用户报告，在一个已经混乱的会话里，AI 会开始重复已经明确排除的方案。

### 错误三：忽视上下文容量警告

不少工具在接近上下文上限时，会通过明显提示、响应质量下降或自动压缩前的征兆发出信号。很多用户选择忽略，继续在同一会话里工作。

**真实后果**：当上下文达到极限时，系统会强行截断早期的重要对话（包括最初设定的架构原则和约束条件），导致不可预测的"失忆症"。AI 会开始遗忘项目的核心规范，产生幻觉。

---

## 背后的原理

### Context Rot（上下文腐烂）

这是一个有专门名称的现象。随着会话的深入，上下文中积累了：
- 被放弃的旧方案（但 AI 仍然"看得见"）
- 冗长的错误日志（占用大量 Token 但信息密度极低）
- 反复尝试的中间代码（互相矛盾的逻辑片段）

**类比**：想象你在一张白板上工作，但每次写错了不能擦掉，只能在旁边写"这个不对"。白板很快就会被各种划掉的内容填满，真正有用的信息被淹没在噪音里。

### 成本的二次方增长

上下文是累积的。如果前 10 轮对话积累了 10 万个 Token，那么发送第 11 轮提示时，系统依然会将前 10 万个 Token 作为负载发送给模型重新计算。

**数学上**：一个 N 轮的会话，总 Token 消耗 ≈ N² × 平均每轮 Token 数。会话越长，每一轮的边际成本越高。

### 负向 Few-shot 学习

LLM 会从上下文中的所有信息中学习模式。当上下文里充满了错误的尝试和失败的推理过程时，模型实际上在进行"负向 Few-shot 学习"——它学到的是"在这个项目里，这类问题的答案是这样的（错误的）"。

这解释了为什么一个已经混乱的会话，即使你明确说"不要这样做"，AI 也会反复犯同样的错误。

## 延伸的习惯

### 事不过三法则（The Third Correction Rule）

如果 AI 连续三次未能正确理解意图，这是上下文已经被严重污染的信号。此时最节省时间的做法是：
1. 撤销工作区中不正确的代码更改（优先用 `git restore --source=HEAD --worktree <path>`、Cursor checkpoint 或等价的精确回退）
2. 关闭当前会话
3. 重新梳理思路，优化初始提示词
4. 开启全新会话

一个拥有更优质初始提示词的干净会话，其输出质量永远优于一个积累了大量纠错历史的冗长会话。

### 重构后立即重置

当你在会话中途对代码库进行了大规模重组（移动文件、重命名模块、修改核心接口），AI 上下文缓存中原有的文件结构和逻辑路径已经完全失效。继续在原会话中工作，AI 会基于过时的信息进行推理。

**具体信号**：AI 开始引用已经不存在的文件路径，或者建议修改已经被重命名的函数。

---

## 个人 / 协作注意

**个人使用**：会话重置的成本很低（几秒钟），但很多人有"不舍得"的心理——觉得之前的对话历史是有价值的。实际上，真正有价值的信息应该在代码里（已提交的 commit）和规则文件里（如 `CLAUDE.md`、`AGENTS.md`、`.cursor/rules`），而不是在会话历史里。

**团队协作**：如果团队成员共享同一个 AI 工具账号或配置，需要约定会话重置的时机。特别是在使用显式 handoff 文件或 Claude Code auto-memory 时，一个人的混乱会话可能会污染团队共享的工作记忆。建议每个人使用独立的本地配置和清晰的交接文件。

---

## 延伸调查方向

以下场景值得进一步深挖，可以作为下一轮内容扩展的方向：

1. **量化 Context Rot 的临界点**：在不同规模的项目中（10 个文件 vs 100 个文件），上下文腐烂从哪一轮开始显著影响输出质量？有没有可测量的指标？

2. **工具差异**：不同工具对上下文腐烂的抵抗能力是否不同？Aider 的 repo map 机制是否能延缓腐烂？Claude Code 的 `/compact` 命令实际压缩了多少有效信息？

3. **会话长度与成本的实测关系**：在真实项目中，10 轮会话 vs 50 轮会话的总 Token 消耗差异是多少？是否真的呈二次方增长？

4. **"三次纠错"的最优阈值**：三次是经验值还是有数据支撑？不同任务类型（Bug 修复 vs 新功能开发）的最优阈值是否不同？
