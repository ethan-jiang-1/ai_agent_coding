# 10 多智能体分流：先做职责隔离，再做并行

## 核心习惯

多智能体的重点不是“多开几个代理”，而是把探索、实现、验证这些角色拆开，并保持拆分有界。

更稳的默认路径是：

- 主线程负责收口
- 子代理优先做只读探索或边界清楚的子任务
- 没有清楚职责时，不升级并行

## 在 Codex CLI 里怎么做

- 真正需要并行探索时，可以让 Codex 使用 subagents。
- 内置的 `explorer` 更适合读多文件找线索，`worker` 更适合执行修改。
- 如果任务更适合保守自动化，就用多个 `codex exec --ephemeral` 加 `-o` 文件串联。
- 如果你已经在交互式会话里，`/agent` 和 `/review` 是最直接的会话内入口；如果这些都不够，再回到 shell 外工作流。

当前 Codex 已经支持 subagents，但这不意味着所有任务都该自动拆代理。更稳的原则是：

- 读多、改少的任务，更适合独立探索角色
- 要改文件的任务，必须明确责任边界
- 并行写入时，写入范围不能重叠
- 如果需要真正的文件系统隔离，subagents 不够，要转到外部 Git worktree；见 `13_advanced_agentic_coordination.md`

## 什么时候不用多智能体

- 任务范围很小
- 只有一个文件要改
- 需求还没定清
- 你自己都说不清每个代理该干什么

这时多智能体只会放大混乱和成本。

## 不要这样做

- 不要把多个会写文件的任务并行打到同一块代码上。
- 不要把“fork 一个旧会话”当成“独立审查者”。
- 不要让代理职责交叉，比如两个代理都负责“顺便修一下相关问题”。
- 不要把 still-under-development 的 fan-out 想象成默认稳定能力。
- 不要在主线程还没形成计划前，就先大规模拆 subagents。

## 验收标准

- 每个代理或每个批次只有一个清楚职责。
- 并行部分主要是只读探索，或者写入范围明确不重叠。
- 最终整合结果可以追溯到哪个代理或哪份 handoff 产出。
