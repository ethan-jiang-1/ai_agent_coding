# 03 先规划，后执行：把读和写强制分开

## 核心习惯

在 Codex CLI 里，规划阶段和执行阶段最好用不同的运行形态表达出来，而不是只靠提示词说“先别改代码”。

## 在 Codex CLI 里怎么做

- 规划阶段，用 `codex exec -s read-only --ephemeral`。
- 规划结果落到文件，而不是只留在屏幕输出里。
- 执行阶段，再切到 `-s workspace-write`，按计划实施。
- 如果你已经在交互式会话里，可以用 `/new`、`/clear`、`/compact` 辅助控制规划节奏；但它们不能替代计划落盘。

最稳的做法是：

1. 只读分析
2. 计划落盘
3. 人看一眼计划
4. 可写执行
5. 验证结果

## 好计划至少要写清楚什么

- 受影响的文件
- 每个文件为什么要改
- 哪些文件明确不改
- 执行顺序
- 风险点和验证方式

如果计划文件连“改哪些文件”都说不清，那还不配进入执行阶段。

## 不要这样做

- 不要一开始就给可写权限，再在提示词里说“先分析一下”。
- 不要让计划只存在会话历史里。
- 不要把超过 3 个文件的改动当成“边写边想”的轻任务。

## 验收标准

- 已经有一个可检查的计划文件。
- 计划和执行是两个可区分的阶段。
- 执行提示词明确引用了计划文件，而不是依赖 AI 自己“记得刚才说过什么”。
