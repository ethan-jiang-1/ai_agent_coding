# 07 模型路由：按任务复杂度选模型，不按情绪选模型

## 核心习惯

Codex CLI 没有 Aider 那种内建三模型分工，所以模型路由主要靠你手动选。重点不是“永远上最强”，而是让强模型只负责真的难的任务。

## 在 Codex CLI 里怎么做

- 跨文件理解、架构决策、复杂重构，用更强的远程模型。
- 常规实现、范围清晰的修改，用默认或中档模型。
- 简单日志分析、格式化、重命名、低风险局部修改，才考虑 `--oss` 本地模型。
- 如果你已经在交互式会话里，优先用 `/model` 切换模型，而不是退出会话后重起一条命令。

一个简单判断：

- 需要跨模块理解吗
- 需要业务推理吗
- 需要很高的正确率吗

如果三项里有两项以上是“要”，就不要为了省一点成本把任务压给弱模型。

## 对 Codex CLI 的特殊点

- `-m <model>` 是显式切换。
- `--oss` 是本地模型入口，不是“便宜版万能替代”。
- 如果要做更细的模型分工，通常要借助 subagents 或不同运行批次，而不是指望 CLI 自动分层。

## 不要这样做

- 不要把 `--oss` 用在大规模重构或安全敏感改动上。
- 不要因为任务“看起来只有一个文件”就默认它是轻任务。
- 不要把模型选择写成硬编码教条，比如“变量重命名一定是轻任务”。

## 验收标准

- 你能解释当前模型选择和任务难度的对应关系。
- 本地模型只承担低风险、低耦合任务。
- 高风险任务的模型选择是显式的，不是默认碰运气。
