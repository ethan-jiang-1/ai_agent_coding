# 09 跨会话状态传递：会话继续是一层，handoff 文件是另一层

## 核心习惯

Codex CLI 的跨会话状态传递，至少分三层：会话历史、项目规则、显式 handoff 文件。长任务不要只靠其中一层。

## 在 Codex CLI 里怎么做

- 同一逻辑任务继续推进，用 `resume`。
- 想基于同一历史走另一条路径，用 `fork`。
- 需要跨人、跨天、跨工具传递时，用 handoff 文件。
- 如果任务已经变成 detached long-running，再考虑 `codex cloud`，不要硬把一切都塞进本地 `resume`。

一个实用优先级：

- 短期续跑，优先 `resume`
- 分叉方案，优先 `fork`
- detached 长程执行，优先 `codex cloud`
- 可审查、可交接、可回看的状态，优先文件

## 为什么 handoff 文件仍然重要

因为会话历史再方便，也有几个问题：

- 别人不一定看得到
- 关键状态不一定一眼能查到
- 过长历史会继续污染后续上下文

文件化 handoff 的好处是显式、可改、可提交、可审查。

## 不要这样做

- 不要把“能 resume”理解成“不需要写 handoff”。
- 不要把一次成功分析只留在屏幕输出里。
- 不要默认非交互式历史和交互式分叉能力完全对等。
- 不要把 `resume` 和 `codex cloud` 写成同一种延续机制。

## 验收标准

- 当前任务的下一步可以被别人接住。
- 至少有一个明确的计划文件、摘要文件或 handoff 文件。
- 你能说清楚这次是靠 `resume` 延续，还是靠文件传递。
