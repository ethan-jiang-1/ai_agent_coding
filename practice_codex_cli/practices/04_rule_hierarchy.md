# 04 规则分层：轻全局，精按需

## 核心习惯

Codex CLI 的规则载体是 `AGENTS.md`。规则设计的目标不是“写得越全越好”，而是“只让当前任务真正需要的规则进上下文”。

## 在 Codex CLI 里怎么做

- 把稳定、跨任务的共识放到根目录 `AGENTS.md`。
- 把只对某个子目录有效的规则，放到更深层目录的 `AGENTS.md` 或 `AGENTS.override.md`。
- 快变的任务说明、一次性的迁移计划，不要塞进 `AGENTS.md`。
- 在交互式会话里，`/init`、`/config`、`/status`、`/memory` 可以帮助你观察和初始化当前规则/配置环境，但它们不等于 `AGENTS.md` 本身。

Codex 会在启动时构建指令链。离当前工作目录越近的规则，优先级越高。

## 该放进 AGENTS.md 的内容

- 稳定的测试和验证要求
- 包管理器、构建命令、提交前检查
- 明确不能碰的目录或约束
- 长期存在的代码风格或协作规则

## 不该放进 AGENTS.md 的内容

- 本周这个特性怎么拆
- 临时调试步骤
- 只对一次任务有效的实施计划
- 大段背景材料或随手笔记

## 不要这样做

- 不要以为 `CLAUDE.md` 会被 Codex CLI 自动读取。
- 不要把所有团队知识都堆在仓库根的一个长文档里。
- 不要频繁重写 `AGENTS.md` 去承载快变任务。

## 验收标准

- 根规则文件短、稳、跨任务有效。
- 特定目录的特殊规则靠路径分层解决，不靠全局长文。
- 任务级信息已经落到计划文件或 handoff 文件，而不是污染规则层。
