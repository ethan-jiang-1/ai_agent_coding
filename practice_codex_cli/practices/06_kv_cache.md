# 06 稳定前缀：让慢变规则稳定，让快变任务外化

## 核心习惯

就算你看不到底层缓存命中细节，稳定前缀仍然有价值。对 Codex CLI 来说，最重要的稳定前缀通常就是 `AGENTS.md` 和固定的工作流骨架。

## 在 Codex CLI 里怎么做

- 让 `AGENTS.md` 保持慢变，不要把任务描述写进去再频繁改。
- 把任务计划、handoff、分析摘要单独放文件。
- 会话一旦变长，就主动压缩任务边界，而不是指望历史无限增值。

这章真正要避免的是：

- 规则层天天改
- 任务层又不落盘
- 导致每次都重新解释一遍项目规则和当前目标

## 对 Codex CLI 的现实理解

- 当前公开资料没有把 Codex CLI 的缓存命中机制讲得像 API 文档那样透明。
- 但官方明确了 `AGENTS.md` 每次启动都会重新构建指令链。
- 所以实践上，稳定规则层、外化任务层，仍然是最稳的做法。

## 不要这样做

- 不要把“缓存”理解成“长会话一定更划算”。
- 不要为了一个短期任务去重写全局规则。
- 不要把长会话里的所有历史都当成资产。

## 验收标准

- 规则文件明显慢于任务文件变化。
- 当前任务的关键状态已经在计划文件或 handoff 文件里。
- 当会话变长时，你有重置或拆分的机制，而不是一直硬撑。
