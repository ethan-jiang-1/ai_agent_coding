# 12 交互控制层：会话控制优先用 slash commands，不要污染主提示词

## 核心定位

这一章不是收容“没地方放的命令”，而是明确 Codex CLI 的一层真实控制面：

- 会话控制
- 交互式观察
- 旁路动作

这些动作的重点不是产出内容，而是控制当前会话，所以不该混进主提示词。

## 这一章为什么存在

这次对 Codex CLI 做 command-surface mapping 之后，发现有一批 slash commands：

- 明显属于交互式会话
- 对真实使用体验有价值
- 但很难和现有外部命令形成清晰的一一映射

对这类能力，强行塞回前面章节只会让结构变乱；但如果继续把它们写成“只是方便小命令”，又会低估 command surface 的行为层意义。

## 当前收录的代表

- `/apps`
- `/copy`
- `/feedback`
- `/sandbox-add-read-dir`
- `/statusline`

这些命令更像：

- 会话内控制层
- 交互专属的小工具层
- 不适合回写成 shell 子命令映射的能力

而不是外部命令的替代品。

## 应该怎么理解它们

- 它们属于交互工作流增强层
- 它们对日常使用重要，但不是原有 11 个习惯的核心骨架
- 它们更适合在已经进入交互式 TUI 之后使用
- 产出型请求走提示词，会话控制型请求优先走命令
- `/ps` 这类与长程运行观察直接相关的命令，不放这一章，而放到第 `13` 章

## 不要这样做

- 不要因为它们重要，就把它们误当成新的通用习惯。
- 不要把所有没地方放的内容都往这里扔。
- 不要忽略它们和真实交互体验之间的关系。
- 不要把会话控制动作混进实现 prompt 里再指望模型自己分层。
