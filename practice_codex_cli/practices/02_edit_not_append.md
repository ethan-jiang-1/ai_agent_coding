# 02 纠错重试：修改原意图，而不是无限追加

## 核心习惯

当方向错了，应该回到“原问题”重做，而不是在污染过的上下文里不停追加补丁说明。

## 在 Codex CLI 里怎么做

- 非交互式 `exec` 走偏时，先回滚文件，再用更好的提示词重新跑。
- 交互式会话想试另一条路径时，用 `codex fork` 分叉，而不是在原会话上越补越乱。
- 如果错误实现已经进入上下文，不要默认继续 `resume`；很多时候直接新开一条更干净。

这章在 Codex CLI 里的落点不是“神奇的撤销按钮”，而是：

- 文件状态回到干净处
- 会话状态回到干净处
- 提示词本身被改好

## 什么时候可以继续追加

只有在目标不变、只是补充一个漏掉的约束时，才适合继续当前会话，例如：

- “再补一个边界条件”
- “按刚才的计划，只实现第一步”
- “不要动测试，只改中间件”

如果已经变成“换一种实现方向”，那就不是追加问题，而是重开问题。

## 不要这样做

- 不要把三四轮失败实现都留在同一个上下文里，然后指望最后一轮突然变好。
- 不要把 `fork` 当回滚工具。`fork` 解决的是分叉，不是恢复文件。
- 不要默认用更长的提示词覆盖更差的上下文。

## 验收标准

- 如果重试，这次提示词比上一次更明确。
- 如果分叉，原会话和新方向是可区分的。
- 当前上下文里没有一长串已经被证明错误的实现历史。
